# 23 · 和工程师的新协作语言——4 个高频场景剧本 + PM 看 trace 最小集

> 第一季《AI 时代 PM 工作流重构》· 第 23 篇 · 第六部分 协作
> 预估字数：5000

---

第二十二个"不以为然"——

**90% 的 PM 跟工程师对 AI 需求还在说「按钮跳转」**。

不是 PM 不愿意学新词。是大多数 PM **以为"PM 不需要懂 prompt 和 eval，那是工程师的事"**——结果每次需求评审，PM 描述完工程师都得返工 3 轮——**因为 PM 不懂协作的边界，提的需求要么"AI 做不到"，要么"AI 完全可以但 PM 没想到"**。

读完这一篇你会知道——**PM 不需要懂 prompt 语法，但必须在 4 个高频协作场景里说对话**：

- 需求评审会
- Eval 结果分歧
- 上线 GO / NOGO
- 线上 Bad Case 紧急

而且——必须知道 PM 自己该跑哪些实验、工程师跑哪些、**半懂比不懂更危险**的 3 个事故模式。

最后给你一个全专栏反复说但从没明说的——**PM 看 trace 的最小集**。

## 上半篇 · workflow

### 4 个高频协作场景的话术剧本

这一节是终极工具——**评审会 / 评估冲突 / 灰度决策 / 紧急 bad case**——4 个场景每个给"对 vs 错"两套话术。背熟即用。

#### 场景 1：需求评审会

**❌ 旧 PM 错的说法**：
> 「能不能给我做一个智能助手，帮用户写文案。」

工程师反应：黑脸。"智能助手"覆盖范围太广 / "写文案"标准不清 / "用户"指谁不明 → **返工 3 轮起步**。

**✅ 新 PM 对的说法**：
> 「我要做一个营销文案生成器。**输入是【产品名 + 卖点 3 条 + 目标受众】，输出是 3 条小红书风格文案（每条 ≤ 100 字 + 1 个 hashtag）**。
>
> Eval 案例我准备了 50 条（已发飞书）。**通过率 ≥ 85% 算上线门槛**。
>
> P99 延迟 ≤ 5 秒。**单次成本 ≤ 0.3 元**。
>
> Bad case 我定义了 4 级：第 4 级（生成违规内容）必须 100% 拦截。」

工程师反应：3 分钟就能决定可行性 + 大概排期。

**核心**：PM 说**业务标准**（输入输出 / 通过率 / 延迟 / 成本 / bad case 等级）—— 不要说 prompt / 模型 / 参数。**那是工程师的事**。

#### 场景 2：Eval 结果分歧

**❌ 旧 PM 错的说法**：
> 「Eval 通过率 92%，可以上线了。」（看到一个数就拍板）

**✅ 新 PM 对的说法**：
> 「Eval 通过率 92%——**但我要看分布**。
>
> 那 8% 不通过的 case 是哪几类？我要知道：
> 1. **失败 case 集中在哪个细分场景**（举例：营销文案在'B 端 SaaS'场景准确率 60%，'C 端美妆'场景 99%）
> 2. **失败 case 的严重度分布**（第 3 级以上比例 > 1%？）
> 3. **是否覆盖了用户最高频的 top 20% 场景**
>
> 如果失败集中在小众场景 → 可以上线 + 灰度。
> 如果失败集中在核心场景 → **不能上线**，先 fix prompt / RAG / fine-tune。」

工程师反应：被尊重的感觉——PM 跟我用同一套语言讨论 eval 分布，不是甩"通过率 92%"了事。

#### 场景 3：上线 GO / NOGO

**❌ 旧 PM 错的说法**：
> 「老板催，今天上。」

**✅ 新 PM 对的说法**（拿第 19 篇 5 大类 pre-launch eval checklist 对照）：
> 「我们走 5 大类 pre-launch eval：
>
> ✅ 类别 1 核心 case：通过率 92%（达标）
> ✅ 类别 2 性能：P50 1.2s / P99 4.8s（达标）
> ⚠️ 类别 3 成本：单次 0.45 元（超预算 0.3 元的 50%）
> ❓ 类别 4 对抗：还没跑红队 eval
> ❌ 类别 5 回归：历史 case 退化 7%（超过 5% 阈值）
>
> **类别 4 没跑 + 类别 5 退化 = NOGO**。
>
> 老板那边我去沟通——给他看 OpenAI 4-29 GPT-4o sycophancy 全量回滚的代价，他会同意延 3 天。」

工程师反应：松一口气——终于有 PM 替我顶住老板压力了。

#### 场景 4：线上 Bad Case 紧急

**❌ 旧 PM 错的说法**：
> 「这个 case 出错了，赶紧改 prompt 修复。」（参见第 21 篇修 A 坏 B）

**✅ 新 PM 对的说法**：
> 「线上出了一个 bad case——我先看 trace 判断属于哪一类：
>
> - hallucination（编造事实）？→ 加 RAG grounding，不要改 prompt
> - refusal（该答不答）？→ 调 safety filter
> - format error（JSON 解析失败）？→ structured output
> - off-topic（偏题）？→ 改 system prompt 边界
> - safety violation（违规内容）？→ 加 content filter
>
> **归类完了再决定修哪个层**。修完跑 regression eval——历史 case 退化 > 5% = 修复失败。
>
> 如果是第 4 级 bad case（公关风险）—— **立刻触发降级到规则**，不要等修。」

工程师反应：PM 终于把 bad case 当成 5 类不同问题处理，不是甩"改 prompt"了事。

### PM 该跑哪些实验 / 工程师跑哪些——边界划分

| 实验类型 | PM 跑 | 工程师跑 | 一起跑 |
|---------|-------|---------|------|
| 评估"这个需求能不能做" | ✅（Claude Code 跑 prototype）| | |
| 写 system prompt | | ✅ | |
| 写 eval case（业务标准）| ✅ | | |
| 写 LLM judge prompt | | | ✅（PM 出标准，工程师写）|
| 跑 batch eval | | ✅ | |
| 看 trace（业务角度）| ✅ | | |
| 看 trace（技术角度，调 prompt）| | ✅ | |
| 处理 bad case（归类 + 决定修哪层）| | | ✅ |
| 灰度决策（基于业务影响）| ✅ | | |
| 灰度技术配置 | | ✅ | |
| Pre-launch eval（类别 1 + 4 业务部分）| ✅ | | |
| Pre-launch eval（类别 2 性能 + 类别 3 成本 + 类别 5 回归）| | ✅ | |
| Cost 给 CFO 算账 | ✅ | | |
| Cost 技术优化（caching / batching / 模型选择）| | ✅ | |

**核心边界**：PM 跑**业务相关 + Claude Code prototype**；工程师跑**技术实现 + 批量 eval + prompt 工程**。

边界划清——**协作摩擦从 80% 降到 20%**。

## 下半篇 · engineering reality

### PM 看 trace 的最小集（全专栏第一次明说）

全专栏反复说"PM 每天看 20-50 条 trace"——但 trace 到底怎么看？这里给最小集。

#### 工具
- **Helicone**（最简单，一行 proxy 接入）
- **LangSmith**（LangChain 官方）
- **Braintrust**（eval + trace 一体化）
- **Arize**（企业级）

公司没接入的话——**自己用 ChatGPT 内置的 conversation export 也能看**。先训练嗅觉，再要求公司接 dashboard。

#### PM 看 trace 必看的 5 个字段

| 字段 | 看什么 | 嗅觉建立 |
|------|-------|---------|
| `input` | 用户原始输入 | 这是不是真实业务场景？还是边角 case？|
| `system_prompt` | 当前生效的 prompt | prompt 边界覆盖了用户场景吗？|
| `output` | AI 的最终输出 | 准确？完整？格式对？|
| `latency` | 首 token 延迟 + 总耗时 | 用户感知慢不慢？|
| `cost` | 单次调用 token 用量 | 单用户单日如果调 100 次会爆吗？|

#### PM 看 trace 的「30 秒判断法」

每条 trace 看完 30 秒内回答 3 个问题：

1. **这条结果用户会满意吗？**（0-5 分主观打分）
2. **如果不满意——是哪一类失败模式？**（hallucination / refusal / format / off-topic / safety）
3. **这是孤立的 case 还是同类反复？**（参见第 21 篇 bad case 传染性）

30 秒判断完——记下来 → 凑够 20-30 条同类的 → 找工程师讨论修复方向。

**核心**：PM 看 trace 不是"读懂技术日志"——是"训练业务 vs AI 输出的对齐感"。

### 半懂 PM 的 3 大事故模式

「**全不懂**」工程师会兜底。「**半懂**」PM 自信瞎指挥——这是最危险的。

#### 事故 1：PM 「我看 ChatGPT 能做，为什么我们不能」

PM 看到 ChatGPT 能做某个 demo，要求自己产品也能做。

工程师解释："ChatGPT 用了 RAG + tool use + reasoning"——PM 觉得"你在找借口"。

**3 周后产品上线**——发现自己的 RAG 没数据、没 tool use、没 reasoning。**最后被迫推翻重做**。

**根因**：PM 半懂 demo，不懂 demo 背后的 infrastructure。

**避免方法**：PM 看到任何 AI demo，先问自己"**ChatGPT 调了哪些 infrastructure（RAG / tool / reasoning）？我们公司有这些吗？没有要先建什么？**"

#### 事故 2：PM "你 prompt 改一下就好了"

bad case 出现——PM 第一反应"改 prompt"。

工程师反复解释："改 prompt 会影响其他 case"——PM 觉得"工程师在拖延"。

**强行改 prompt 后**——其他 5 个场景退化（参见第 21 篇修 A 坏 B）。

**根因**：PM 半懂 prompt 是"指令"，不懂 prompt 是"系统级配置"。

**避免方法**：bad case 出现先归类（5 类）→ 决定修哪一层（详见第 21 篇 5 步），prompt 是**最后选项**不是第一反应。

#### 事故 3：PM "我看这个 eval 通过率 95%，可以上线了"

PM 看 eval dashboard 显示"通过率 95%"——拍板上线。

上线后崩——因为：
- 那 5% 不通过的 case 是**核心 case**
- 5% 的 case 影响 30% 的用户
- 没看分布只看均值

**根因**：PM 半懂 eval 是"打分"，不看分布。

**避免方法**：永远看分布不看均值——拿场景 2 的话术对工程师讨论。

### PM 该懂什么 vs 不该懂什么

| 必须懂 | 不需要懂 |
|--------|---------|
| 4 个高频场景的话术 | Prompt 的具体写法（工程师的活）|
| 看 trace 5 字段 + 30 秒判断法 | Eval 框架的实现（promptfoo / DeepEval 代码）|
| Bad case 5 类归类 | 模型训练原理 |
| 4 类降级策略框架 | RAG 向量检索算法 |
| 4 类 AI 监控指标 | 模型推理优化 |
| Pre-launch 5 大类 checklist | LLM 内部 attention 机制 |
| 5 数字给老板汇报 | Tokenizer 的具体实现 |

**核心**：PM 要懂"协作场景需要的概念"，不需要懂"工程实现的细节"。

## 业内对 PM 角色变化的共识

### 共识 1：PM 必须能自己跑 Claude Code prototype

详见第 17 篇——这是 PM 在 4 个协作场景前的"前置作业"。

### 共识 2：PM 必须每天看 trace

详见上面"PM 看 trace 最小集"——这是协作的"嗅觉训练"。

### 共识 3：PM 必须有「工程感」

什么是工程感？知道：
- 这件事情大概多久能做完
- 这件事情大概多贵
- 这件事情会引入什么风险
- 哪些事情 AI 做不到（物理边界）

没有工程感的 PM = AI 时代被淘汰的 PM。

## 质量工程视角看 PM-工程师协作 = 共同语言定义系统边界

**8 年做软件质量工程的视角**——

质量工程的核心方法是「**FMEA**（Failure Mode and Effects Analysis）」——失败模式与影响分析。

FMEA 要求**业务方和工程师用同一套语言**讨论：
- 哪些失败模式可能发生
- 每种失败的影响多大
- 哪些是可接受的（让它发生但兜底）
- 哪些是不可接受的（必须 block）

**AI 产品的 PM-工程师协作本质就是共同做 FMEA**——
- PM 提供"业务影响视角"
- 工程师提供"技术可行性视角"
- 共同语言：4 个高频场景剧本 + bad case 5 类 + 5 大类 pre-launch eval

PM 不懂这套语言 = 无法参与 FMEA = 让工程师独自承担产品决策责任。

**90% 的 PM 还在说"按钮跳转" = 把 AI 产品的责任全推给工程师**。
