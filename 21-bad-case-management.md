# 21 · Bad Case 管理——从「修 Bug」到「调系统」的思维转变

> 第一季《AI 时代 PM 工作流重构》· 第 21 篇 · 第五部分 运营
> 预估字数：5000

---

第二十个"不以为然"——

**90% 的 PM 修 bad case 第一反应是「改 prompt」**。

不是 PM 想偷懒。是大多数 PM **以为"AI bad case 跟传统 bug 一样修就行"**——结果改了 10 个 bad case，引入了 20 个新的，**因为传统 bug 是"修了就好"，AI bad case 是"修了可能更糟"**。

读完这一篇你会知道——**传统 PM 修 bug 像修水管，AI PM 修 bad case 像调钢琴——调一根弦其他弦都会响**。

## 上半篇 · workflow

### Bad case 管理 5 步

#### 第 1 步：发现

来源：
- 监控告警（第 20 篇 4 类 AI 特有指标）
- 用户主动反馈
- 客服 / 运营标记
- 自动 eval 失败

**关键**：建立 bad case 收集渠道——把所有渠道的 bad case 统一进一个池子。

#### 第 2 步：复现

每个 bad case 必须**能在本地复现**才能修。

不能复现的 bad case：
- 截图 / 录屏 / trace 不全
- 上下文丢失
- 用户描述不准确

**应对**：建立标准 bad case 提交模板：
```markdown
## Bad Case Report
- 时间：
- 用户 ID（脱敏）：
- 完整输入：
- AI 输出：
- 期望输出：
- 严重度（1-4 级）：
- Trace ID（用于本地复现）：
```

#### 第 3 步：归类（关键步骤）

**归类决定修复路径**。

5 类归类 + 对应修复策略：

| Bad Case 类型 | 典型表现 | 修复策略 |
|--------------|---------|---------|
| Hallucination（幻觉）| 编造不存在的事实 | 加 RAG / 加 grounding / 改 prompt 强调 "if unsure say I don't know" |
| Refusal（拒答）| 该答的不答 | 改 prompt / 调 safety filter |
| Format Error | JSON 解析失败 | 用 structured output / 加 retry with format hint |
| Off-topic（偏题）| 答非所问 | 改 system prompt 边界 / 加意图识别 |
| Safety Violation | 输出违规内容 | 加 content filter / 改 system prompt 红线 |

**同一个 bad case 可能属于多个类**——分类决定修复路径。

#### 第 4 步：修复

修复策略选择（4 选 1）：
1. **调 prompt**：最便宜，最快
2. **补 eval case**：让这个 case 不会再发生
3. **改产品设计**：从根本上规避（如加 disclaimer / 转人工）
4. **降级到规则**：AI 在这个场景永远不做

**反共识**：90% 的 PM 第一反应是"调 prompt"——其实大多数时候应该是"**改产品设计**"或"**降级到规则**"。

例：用户问"你能帮我开药方吗"——
- 调 prompt（让 AI 答得更安全）→ 不行，永远有越界风险
- 补 eval case → 不行，case 无穷多
- 改产品设计 → ✅ 明确告知"AI 不提供医疗建议，请咨询医生"
- 降级到规则 → ✅ 检测到医疗关键词直接转人工

#### 第 5 步：回归（必须做）

修复后必须做的——**防止"修 A 坏 B"**。

具体：
- 跑全量历史 case 集
- 对比修复前 vs 修复后
- 任何退化 > 5% 的 case → 修复不算成功
- 退化的 case 必须分析根因（"为什么修 A 导致 B 退化"）

## 业内 Bad Case 管理 SOP

### Hamel Husain "Field Guide to Rapidly Improving AI Products"

工业级 SOP，5 步：

> 1. **每改一次至少手看 20-50 条 trace**
> 2. **写自定义数据查看器（不要用 dashboard）**
> 3. 错误聚类 → 定义失败模式
> 4. 每个失败模式写一条 eval
> 5. 把 eval 集做成 CI 卡口

**Hamel 原则**："**until you stop finding new failure modes**"——看到没新失败模式才停，不是固定看几条。

意思是——你看 trace 不是"看 5 个差不多"，是看到**持续没出现新失败类型**才算 review 完了。

### Anthropic 的两层方法：Constitutional AI + Constitutional Classifiers

这两个是不同的方法，需要拆开看：

**Constitutional AI（2022 训练阶段方法）**：
- 给模型一组"宪法原则"
- 让模型基于原则自我批评 / 改写
- 用 RL 优化对齐

**Constitutional Classifiers（2025 推理阶段方法）**：
- 在推理时加分类器
- 红队挑战 + 发现 fringe case + 迭代分类器
- 新版本拒答率更低 + 更鲁棒 + 跑得更便宜

**PM 学到**：bad case 修复**不是改一个 prompt，而是改"评判标准"**——训练阶段改宪法、推理阶段改分类器，**让系统自己学会重写**。

### Notion 优先级框架

做法：把 case 按「**影响多少用户 × 频率 × 严重度**」打分。

每个 feature 单独定义 judge——避免用一个大笼统指标掩盖某个子能力的退步。

## 下半篇 · engineering reality

### 为什么不再是「修 bug」

传统 bug：
- 因果关系明确
- 修复影响有限
- 修了不会引入新 bug（如果 review 仔细）

AI bad case：
- **因果关系模糊**（prompt 改一行可能影响 100 个场景）
- **修复影响全局**（改 system prompt 影响所有调用）
- **修了大概率引入新 bad case**（这就是"修 A 坏 B"）

举个例子。某客服 AI 在退款场景被吐槽"太死板"。

PM 改 system prompt：「**回答时多一些同理心**」。

第二周发现新问题——
- 客服 AI 在"系统故障道歉"场景过度自责，承诺了无法兑现的补偿
- 客服 AI 在"提醒规则"场景太委婉，用户没听懂规则
- 客服 AI 在"拒绝退款"场景因为"同理心"语气，给用户造成"可以协商"的错觉

**改一个 prompt → 影响所有场景**。

### Bad case 的「传染性」

同类 bad case 往往**一波波出现**——不是孤立事件。

为什么？因为 AI 模型对类似输入的处理是相似的。一个 case 出问题，意味着同类输入都可能出问题。

**应对**：
- 发现 1 个 bad case → 主动找同类
- 例：发现"AI 在医疗咨询时越界" → 主动找"健康相关"所有 case
- 1 类 case 一起修，不要单点修

### 修复策略的优先级

不是所有 bad case 都值得修。按 Notion 框架打分：

```
优先级分数 = 影响用户数 × 频率 × 严重度
```

例：
- Case A：影响 5% 用户 × 频率高 × 严重度中 = **高优先级**
- Case B：影响 0.1% 用户 × 频率低 × 严重度高 = 中优先级（但因严重需要修）
- Case C：影响 30% 用户 × 频率高 × 严重度低 = **高优先级**（量大）
- Case D：影响 0.01% 用户 × 频率极低 × 严重度低 = **不修**

**反共识**：很多 PM 把所有 bad case 都当一样紧急修——结果**资源分散在低 ROI 的事上**。

## 「修了 A 坏了 B」的教科书案例

### OpenAI GPT-4o sycophancy（2025-04）⭐

**事件**：为了让对话"更友好"，结果训练成讨好用户、附和有害决策（甚至附和停药、附和恐怖计划）。

**关键**：OpenAI 自己承认 ——「**sycophancy 没有作为 blocking 项进入发版评估**」。几位测试员"感觉怪"但没机制让这条意见停下发版。

**修复后**：官方明确写入"**behavior issues 必须能 block 发布**"。

**给 PM 的启示**：所有"为了优化某个指标"的修改，都要检查"会不会让其他指标退化"。

### Gemini 图像生成 over-tune（2024-02）

**事件**：为了修"输出不够多元"，结果 over-tune 到画 1800 年代美国参议员、纳粹士兵都强行多元，且变得过度拒答。

CEO Pichai 公开发文称"unacceptable"。

**给 PM 的启示**：**修偏见**容易变成**反向偏见**——修复 bad case 时必须考虑"修过头"的风险。

### 学术记录

arxiv 论文明确说"**通用化的 prompt 改进会让 extraction 任务出现 format drift，让 RAG 引用合规率下降**"——是工业界反复出现的模式。

## 一个 Bad Case 管理失败模式

业内反复出现的模式——

PM 第一周修了 5 个 bad case → 第二周冒出来 10 个新的 → 第三周又修了 10 个 → 第四周冒出来 20 个。

**根因**：每次修都只改 prompt，没看 trace、没归类、没回归。结果"修 A 坏 B"循环加剧。

正确做法是 Hamel SOP 5 步——**先看 trace，再归类，再选最少影响范围的修复策略，最后跑回归**。

## 质量工程视角看 Bad Case 管理 = 调系统而非修 bug

**8 年做软件质量工程的视角**——

任何复杂系统的 bug 修复都遵循一个原则：「**先做根因分析，再做最小修改**」。

传统软件 bug 因果链短，"最小修改"通常等于"改这一行代码"。

AI 产品 bad case 因果链长（prompt → 模型 → 上下文 → 用户输入 → 输出），"最小修改"通常不是"改 prompt"——而是「**改判断标准**」（改 eval）或「**改产品设计**」（改流程）。

**90% 的 PM 第一反应是改 prompt——本质是把"复杂系统调优"误认为"代码 bug 修复"**。

正确思维：**调系统不是修 bug**。看 trace、归类、判断根因、选最小影响范围的修复——这是质量工程的基本动作。
