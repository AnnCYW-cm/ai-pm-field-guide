# 10 · 技术路径决策树——RAG / Prompt / Agent / Fine-tune 怎么选

> 第一季《AI 时代 PM 工作流重构》· 第 10 篇 · 第二部分 方案
> 预估字数：5000

---

第九个"不以为然"——

**90% 的 PM 一上来就选 RAG / Agent / Fine-tune——跳过 Prompt**。

不是 PM 想搞复杂。是大多数 PM **以为"高级问题需要高级方案"**——结果 3 个月后发现「RAG 调不好」的根因其实是 prompt 设计问题，**$40K + 3 个月烧掉**。

读完这一篇你会知道——**RAG / Prompt / Agent / Fine-tune 不是平行选项，是有优先级的决策树**。选错路径的代价远大于选错模型。

## 上半篇 · workflow

### 4 路径的本质差异

每条路径解决一个不同的问题：

| 路径 | 解决什么问题 | 一句话理解 |
|------|------------|----------|
| **Prompt** | 「指令理解」问题 | 让模型知道你想要什么 |
| **RAG** | 「知识缺失」问题 | 让模型知道你的私有数据 |
| **Agent** | 「流程编排」问题 | 让模型自主完成多步任务 |
| **Fine-tune** | 「风格 / 能力定向」问题 | 让模型按你的风格输出 |

**混淆这 4 种问题 = 选错路径 = 浪费 3 个月 + ¥40K-100K 工程成本**。

### 决策树的提问顺序

按「**最复杂的最后选**」原则倒推：

```
Q1：能用 Prompt 解决吗？
  → 试着改 system prompt + few-shot examples
  → 跑 50 个 case
  → 准确率达标 → 选 Prompt，结束
  → 不达标 → Q2

Q2：是知识缺失吗？
  → 模型回答错是因为"它不知道你的私有数据"吗？
  → 是 → 选 RAG
  → 不是 → Q3

Q3：是流程编排吗？
  → 任务需要多步决策 + 工具调用 + 不可枚举分支吗？
  → 是 → 选 Agent
  → 不是 → Q4

Q4：是风格 / 能力定向吗？
  → 需要模型按特定风格输出？或者需要专属能力？
  → 是 → 选 Fine-tune
  → 不是 → 重新检查需求，可能需求本身有问题
```

### 快速判断口诀

> **能 Prompt 不 Agent，能 RAG 不 Fine-tune**

这两条口诀是基于成本 + 复杂度的：
- Prompt 成本最低（改文字）
- RAG 成本中等（建向量库 + 检索 pipeline）
- Agent 成本高（多步调用 + 工具集成）
- Fine-tune 成本最高（数据准备 + 训练 + 维护）

**60% 你以为要 Fine-tune 的问题，其实 Prompt 就能解决**。

### 混合路径设计

大部分生产级产品是 **2-3 种路径组合**：

```
案例：企业级智能客服

主路径：Prompt + RAG
  - Prompt 定义客服角色、回复风格、边界
  - RAG 检索企业知识库（产品手册、常见问题、订单记录）

辅路径：Agent
  - 用户要求"帮我退款" → Agent 调用退款 API
  - 用户要求"查订单" → Agent 调用订单系统

避坑：不上 Fine-tune
  - 客服话术变化频繁，Fine-tune 跟不上
  - 用 Prompt + few-shot 解决风格问题
```

## 下半篇 · engineering reality

### 误用代价分析

#### 误用 1：Prompt 能解决的硬上 Agent

代价：**多花 5 倍开发成本 + 10 倍调试成本**。

为什么？因为 Agent 是多步串联——每一步都可能崩。Prompt 是单步——只调一次。

调试一个 5 步 Agent 比调试一个 prompt 难 50 倍。每一步都要：
- 看 trace
- 判断哪一步出错
- 决定是 prompt 问题还是工具问题
- 改完跑一遍 5 步全流程
- 改完可能还是错

如果你的需求只是"让模型按格式回答"——**Prompt 改 system prompt 就够了，别搞 Agent**。

#### 误用 2：RAG 解决不了的强上 RAG

3 类问题硬上 RAG 必崩：
- **风格问题**："让模型回答更专业"——这是 prompt 问题，不是知识问题
- **指令问题**："让模型按 JSON 格式输出"——这是 structured output 问题
- **逻辑问题**："让模型推理多步"——这是 chain-of-thought 问题

RAG 只解决「知识缺失」。**你给它装 1 万篇文档也变不出指令理解能力**。

#### 误用 3：Fine-tune 治幻觉（最贵的误解）

**Fine-tune 不能修幻觉**。

这是 IBM / Google Cloud / Anthropic 文档反复强调的反共识。

Fine-tune 改的是"风格 / 格式 / 输出倾向"——它**不能让模型知道新事实**。如果模型不知道某个事实，fine-tune 之后还是不知道。

唯一能让模型"知道新事实"的是 RAG（动态注入知识）或者在线学习（实时更新参数，目前主流模型不支持）。

#### 误用 4：「RAG 就是把文档塞进向量库连模型」

这种实施 **90% 失败**。

原因：
- chunk 设计差（按固定字数切，破坏语义）
- 检索质量差（embedding 模型不匹配领域）
- 生成段落错配（retrieval 出来的片段答非所问）

**做 RAG 不是建向量库，是建一套「chunk + retrieve + rerank + generate」完整 pipeline**。每个环节都要单独 eval。

### 真实误用案例

#### 案例 1：$40K fine-tune 只为治幻觉

某团队花 **$40K fine-tune 模型治幻觉**，事后发现 RAG 层 **$800、2 周就能解决**。

**教训**：先排除 Fine-tune（最贵），再判断 Agent（最复杂），再 Prompt vs RAG。

#### 案例 2：RAG 做了 3 个月发现是 Prompt 问题

业内反复出现的模式——团队上 RAG → 调向量库 → 调 reranker → 调 chunk size → 调 embedding model → 3 个月效果还是不行。

回头一看 system prompt，发现没说清"answer must be grounded in provided context"。

**改 system prompt 1 小时，效果直接达标**。

#### 案例 3：硬上 Agent 的 5 倍成本

一个原本 prompt 能搞定的"客户意图分类"需求，团队上了 Agent——拆成"读输入 → 调分类 API → 调置信度 API → 决策 → 输出"5 步流程。

开发周期从 3 天变成 3 周。调试成本从 5 个 case 变成 50 个 case。**上线后 P99 延迟从 500ms 变成 3s**。

最后回退到 prompt 单步分类。**5 倍成本 + 10 倍延迟，换来同样的效果**。

## 4 路径的「反向适用场景」

| 路径 | 该用 | 不该用 |
|------|------|-------|
| **Prompt** | 60% 你以为要 fine-tune 的问题 | 真的需要新知识 |
| **RAG** | 知识库会变 / 需溯源 / 内部数据 | 问题是"风格不对"而不是"知识不对" |
| **Agent** | 真有多步决策 + 工具调用 + 不可枚举分支 | 流程能写成 if-else 的 |
| **Fine-tune** | 风格 / 格式 / 特定领域语气 | 想让模型记住事实、想修幻觉 |

### Anthropic prompt caching 改变了游戏规则

Anthropic 2024 推出 prompt caching 之后——**Prompt 路径的适用范围被大幅扩展**。

以前你需要 fine-tune 才能让模型"记住"复杂指令——因为 system prompt 太长、每次调用都贵。

现在你可以把超长 system prompt（5K-50K token）放到 cache 里——读 cache 价格只有正常 input 的 1/10。

**含义**：很多原本要 fine-tune 的场景，现在用 Prompt + caching 就够了。

例：让模型按公司风格回邮件 → 把 100 个范例邮件塞进 system prompt + cache → 比 fine-tune 便宜 10 倍 + 调整起来更灵活。

## 4 路径决策表

把所有维度合在一张表：

| 维度 | Prompt | RAG | Agent | Fine-tune |
|------|--------|-----|-------|-----------|
| 解决问题 | 指令理解 | 知识缺失 | 流程编排 | 风格定向 |
| 实施成本 | 低（1-3 天）| 中（2-4 周）| 高（1-3 月）| 极高（3-6 月）|
| 维护成本 | 低 | 中 | 高 | 极高（每次模型升级要重训）|
| 调试难度 | 低 | 中 | 高 | 极高 |
| 适合阶段 | 任何阶段 | 上线后扩展 | 成熟期 | 大规模 + 有专属数据 |
| 失败率 | 低 | 中（pipeline 复杂）| 高 | 极高 |
| 1 人作者推荐度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐ | ⭐ |

**给 1 人 / 小团队的建议**：90% 场景用 Prompt，10% 场景上 RAG，**Agent 和 Fine-tune 几乎不要碰**。

## 质量工程视角看 4 路径决策

**8 年做软件质量工程的视角**——

任何工程问题都应该从「**最简单的方案先验证**」开始。这是 Occam's Razor（奥卡姆剃刀）原则在软件工程的应用——「**如无必要，勿增实体**」。

放到 4 路径决策上——
- 最简单的方案是 Prompt（改文字）
- 最复杂的方案是 Fine-tune（重新训练）

**先验证 Prompt 能不能解决**——
- 能解决 = 省 $40K + 3 个月
- 不能解决 = 这时再考虑复杂方案

这不是"投机取巧"，是工程的基本动作——**避免过早承担不必要的复杂度**。

业内反共识：**60% PM 以为要 Fine-tune 的问题，其实 Prompt 就能解决**。背后的原因不是 Prompt 真能解决一切，是 PM 没认真试过 Prompt 就跳到了 Fine-tune。

## 给你的可执行清单

#### 动作 1：每个新需求按决策树过一遍
- Q1 Prompt？→ Q2 RAG？→ Q3 Agent？→ Q4 Fine-tune？
- 倒推顺序，不要正推

#### 动作 2：把"能 Prompt 不 Agent，能 RAG 不 Fine-tune"贴在工位
- 评审会上随时对照
- 工程师说要 Fine-tune 时 → 反问"Prompt + caching 试过没"

#### 动作 3：3 个常见误用立即排查
- "Fine-tune 治幻觉"→ 改用 RAG
- "RAG 改风格"→ 改用 Prompt
- "Agent 跑 if-else 流程"→ 改用 Prompt

#### 动作 4：评估 prompt caching 适用性
- 你的 system prompt > 1024 token 吗？
- 复用率 > 2 次吗？
- 是 → 立刻上 cache（性价比最高的路径优化）
