# 01 · 你做的还是 PM 吗？—— AI 时代 PM 工作流的 6 个崩塌点

> 第一季《AI 时代 PM 工作流重构》· 第 1 篇 · 序章
> 预估字数：5800

---

8 年做软件质量工程 + 这两年深度参与 AI 产品咨询的观察是——

2026 年 AI PM 圈最扎心的不是「**我不会了**」，是「**我不以为然**」。

写了 8 年 PRD 的资深 PM 觉得「不就是 PRD 嘛，加点 AI 章节呗」——
直到评审会被工程师反问「**你的评估集呢**」，第一次愣住。

带过 5 年团队的产品总监觉得「不就是评审嘛，按流程走呗」——
直到上线第三天 bad case 爆出来，发现"**改 prompt**"改了 3 轮反而越改越糟。

能搞定老板预算的 leader 觉得「不就是成本嘛，工程师会算」——
直到 CFO 拍着账单问「**你们家每月烧多少**」，张口说不出数字。

不是「能力下降」。是「**用传统 PM 的思维做 AI 产品——表面上跑通，实际上埋雷**」。

但更扎心的是——**埋雷只是后果**。
**「不以为然」的真正代价，是被时代用 5-10 倍的速度甩开**。

---

## 紧迫感先摆桌面：5-10 倍效率差是怎么来的

**Lovable**：瑞典创业公司，2024 年 Stockholm 创立。2025-Q3 快照——**18 人公司，~$17M ARR，人均接近 $100 万**，是传统 SaaS 行业基准的数倍（公司在持续扩张中，数字随时间变化）。

**Cursor / Anysphere**：2025-Q3 工程团队约 100 人，撑起 $1B 量级 ARR——**人均收入远高于传统 SaaS**。

**Anthropic**：~2000 人公司里销售 quota carrier 只有**数十名**，服务数十万企业客户——单销售对应客户数远高于传统 SaaS。

（以上数字以公开口径为准，AI 应用层公司增长极快，建议在自家场景验证最新数据。）

这些不是孤例。这是**「AI Native 工作流」的产出**——

- 一个 PM + Claude Code + 2 个 AI Engineer + 1 个 Eval 工程师，**3 个月做出传统团队 12 个月的产出**
- 写 Product Note + 跑 Claude Code 验证，**30 分钟决定要不要立项**——传统流程要 2-4 周
- prompt 改完跑 eval 直接发版，**8 小时迭代一次**——传统 PRD-RD-QA 迭代是 2-4 周

**速度差 10-50 倍**。

David Haberlah 在 Medium 拆解了 638 位 PM 的实际产出——AI 相关内容从 2023 年初的 4% 涨到 2026 Q1 的 67%，**mocking & prototyping 的工作占比从 60-70% 跌到 30-40%**。

腾出来的 30% 时间被什么吃掉了？**评估、agentic 架构、非确定系统的调试**——也就是下面 6 个崩塌点对应的新工作。

**67% 这个数字只是"已经在转的人"**。

更扎心的另一面是——**33% 的 PM 还在用旧方式做 AI 产品，每个项目周期是别人的 5-10 倍**。

---

## 这一切的本质：3 件事同时在变

读完上面你可能想问——为什么差距这么大？是 AI 工具厉害还是新 PM 厉害？

都不是。

是因为**整个 PM 工作的底层，正在同时发生 3 件事的变化**——而 PM 圈对这 3 件事的认知度完全不同。

### 变化 1：思维变了（确定性 → 概率性）⭐ PM 圈基本意识到了

传统软件是**确定性系统**——输入 A 永远输出 B。整个 PM 工作流（访谈、PRD、评审、开发、上线、运营、复盘）全部建立在这个假设上。

AI 产品是**概率性系统**——输入 A 可能输出 B、B'、B''。同一个 prompt 跑 100 次可能 95 次对 5 次幻觉。

Scrum.org 2024 年那篇《Definition of Done for AI Agents》写得最直白：

> "**AI 是概率性的——同一个 prompt 跑 100 次可能 95 次对 5 次幻觉。如果你的 DoD 还是 Pass/Fail 二值检查，你不是在测试 agent，你在赌博。**"

这一层是显性的——PM 圈大多数人知道。

### 变化 2：节奏变了（流水线 → 反馈环）⭐ PM 圈部分意识到了

传统软件团队是 **PM-RD-QA 的流水线**——一棒接一棒：
- PM 写 PRD 给 RD（2 周）
- RD 写代码给 QA（4 周）
- QA 测试给上线（1 周）

每一步有清晰交接、清晰交付物、清晰责任边界。**节奏单位是「周/月」**。

AI Native 团队是 **PM-Prompt-Eval 的反馈环**——三方持续并行：
- PM 写 Product Note + 定 eval（持续）
- Prompt 工程师 / AI Engineer 调 prompt（持续）
- Eval 工程师跑回归 + 看 trace（持续）

每天异步对话。每周看 eval 数据决定走向。**节奏单位从「周」压缩到「天/小时」**。

这一层是半隐性的——意识到的 PM 已经在重塑工作日，没意识到的还在等 RD 排期。

### 变化 3：工具变了（旧 PRD → BMAD / Spec / Claude Code）⭐ PM 圈大多还没看到

这一层最致命。

不是"PM 工具升级"——是 **PM 这个工种的全部交付物形态，正在被一套新方法论彻底重写**：

#### BMAD-METHOD（Breakthrough Method of Agile AI-Driven Development）

把 LLM 拆成多个专业化 Agent 角色（Architect / PM / Developer）——**文档而非代码成为 source of truth**。Developer Agent 发现需求与架构冲突时**必须 halt and report**，不能自主决策。

这意味着——PM 写 PRD 的形态完全要变。**不再是"给人看的文档"，是"给多个 Agent 协同消费的结构化规格"**。

#### GitHub Spec Kit（Spec-Driven Development）

2025-09 开源。核心理念一句话：

> "**Specifications become executable, directly generating working implementations rather than just guiding them**."

——规格本身可执行，**直接生成实现而非指导实现**。

流程是 `/specify` → `/plan` → `/tasks` → `/analyze` → 由 Claude Code / Copilot / Gemini CLI 直接执行。

这意味着——PM 写规格的方式完全要变。**不再是"写完给 RD 读"，是"写完让 Agent 跑"**。

#### Anthropic 的「Product Note」取代 PRD

Mike Krieger（前 Instagram 联创、Anthropic CPO）公开讲——PM 不再写 10 页 PRD，改写 **3-4 段的 Product Note**（用户意图 + 期望结果 + 具体指标），配两个上下文文件：
- `product_area_context.md`（业务规则，PM 维护）
- `code_context.md`（技术现状，工程维护）

Cat Wu（Anthropic Claude Code Head of Product）的名言：

> "**Prototypes over PRDs**."

——能用 Claude Code 做出原型的就不写 spec。

这意味着——**PM 的文档形态从「10 页 PRD」直接压缩到「3 段 Note + 一个跑通的原型」**。

#### Claude Code / Cursor 让 PM 自己跑通验证

**Boris Cherny（Anthropic Claude Code 负责人）公开说**：自 2025-11 起 100% 代码由 Claude Code 写，自己已不手敲。

**Sachin Rekhi（前 LinkedIn）展示**：1500 PM webinar 上演示 13 个 PM 自建 skills——客户访谈合成、NPS 自治、无 SQL 的 EDA、产品战略 critique、release notes 自动化。

这意味着——**PM 不再需要"等工程师才能验证需求"**。30 分钟用 Claude Code 跑通，决定是否立项——传统流程需要 2-4 周排期 + 评审 + 开发。

### 3 层变化同时叠加 = 6 个崩塌点同时崩

| 变化层 | 对应崩塌点 |
|--------|----------|
| 思维变了（概率性）| 崩塌点 1（访谈失效）、崩塌点 5（bad case）、崩塌点 6（复盘归因）|
| 节奏变了（反馈环）| 崩塌点 3（评估集评审）、崩塌点 5（迭代速度）|
| 工具变了（BMAD/Spec/Claude Code）| 崩塌点 2（PRD 写不下去）、崩塌点 4（成本估算）、贯穿所有 |

**3 件事同时变，PM 工作流的 6 个节点同时崩**。

### 在读 6 个崩塌点之前——先把视角抬到 2027

6 个崩塌点是**当下**的崩。但写这本书时心里清楚一件事——**12-24 个月后 PM 工作流会再崩一次，崩的方式完全不同**。

三个已经在发生的信号：

**Context Engineering 正在替代 prompt engineering**。2026 年业内的共识已经是"who controls the agent's context controls its behavior"——PM 的新底层技能不是写 prompt，是设计 agent 的 working memory + tiered storage + 何时该 forget。MemGPT、Letta、Mem0 已经不是研究项目，是生产组件。

**Memory 进入 PM 必考**。Anthropic 2026 年推出的 Dreaming/Orchestration Memory——agent 自己识别"哪些工作流团队反复收敛到一起"。PM 设计的不再是单次对话，是**模型如何跨会话学习你的产品**。

**Agent SDK 改变产品形态**。Anthropic 2026-06 把 Agent SDK 独立计费——意味着"调一次 API"被"派一组 sub-agent 跑 6 小时"取代。Claude Mythos（2026-05）50% horizon 已到 16 小时——**"任务即产品"正在替代"功能即产品"**。

6 个当下崩塌点先读完。但每读一个——心里要带一个问题：**这一条在 2027 还成立吗**？答案大多是"形式还在，内容已变"。这就是为什么 PM 必须**同时跑 L2 当下工作流 + L3 未来视野**——本书 26 篇是 L2 工具箱，**L3 视野本书第 26 篇会给你校准框架**（L2/L3/L4 三层保鲜期）。

---

下面把这 6 个崩塌点摊在桌面上。每一个都是「**旧方式 vs 新方式**」摩擦的具体瞬间。

读完之后——
- 如果你说「这 6 个我都遇到过」——那这个专栏就是为你写的
- 如果你说「这 6 个我都没遇到过」——**先别急着划走**。两种可能：要么你还没真正做 AI 产品，要么你已经在踩坑只是没意识到。第二种最危险。

## 崩塌点一：用户访谈拿不到真需求

这是第一个出现，也是最反直觉的一个。

传统 PM 的训练是：先做用户访谈，再写需求文档，再开发。**访谈是源头**。

AI 时代做用户访谈——做完一轮会发现一件事：**用户告诉你他们想要的 AI，是科幻片里的 AI；他们愿意付钱的 AI，是工具栏里的 AI**。

中间这条鸿沟，访谈跨不过去。

Springer Nature 一篇用户研究综述给了这种现象一个学术名字：「**pre-experience expectations 与 post-experience evaluation 的系统性落差**」——用户没用过你的 AI 之前的预期，和用过之后的评价，根本不是同一件事。

这不是用户不诚实。这是用户的预期来自 ChatGPT、来自 Her、来自 Jarvis。他们坐在你面前，脑子里想的不是你的产品，**是科幻片**。

**Humane AI Pin 是这个崩塌点最贵的样本**。融资 2.3 亿美元、峰值估值 7 亿、用户访谈和原型测试都给出过正向信号——最终 2025 年 2 月卖给 HP 仅 1.16 亿，**返修件数超过销量**。「AI 别在胸前问问题」的需求只在科幻片成立。

Marty Cagan 在 SVPG 2025 年的文章里说得更直接：

> 「Product discovery 是判断（judgement），AI 可以自动化交付，但发现未满足需求、验证价值这件事，AI 不能。」

但他没说的是：**人也越来越难做了**——因为用户的「未满足需求」在 AI 时代里**被科幻片重新定义了**。

**用质量工程视角看**——这是「需求作为概率分布」和「访谈作为单点采样」之间的根本错配。访谈采到的是用户脑子里的科幻预期，模型能交付的是 75% 准确率 + 5% 幻觉的分布。两件事根本对不上。

## 崩塌点二：PRD 写不下去

第二个崩塌点紧接着第一个。

按访谈拿到了一份"需求清单"。打开 Word，开始写 PRD。

写到一半停下来——

- 这个功能的"完成定义"是什么？传统答案是"按预期跑通"。但 AI 输出是概率的，预期是分布。
- 这个交互的"页面跳转"怎么画？传统答案是"点 A 按钮跳 B 页"。但 AI 产品的交互越来越没有"页面"，是对话、是上下文、是 agent 流。
- 这个需求的"验收标准"怎么定？传统答案是"功能列表全部勾完"。但 AI 产品的功能不是离散的，能力是连续的。

写不下去。

不是不会写 Word，是 **「PRD 这个文档形态本身」在 AI 产品上失效了**。

微软 CPO Aparna Chennapragada 在 Lenny 的播客里说得最锋利：

> "**Prompt sets are the new PRDs.**"

不是"PRD 模板要更新"。是 **PRD 这个东西本身正在被替换**。Aparna 说："prompt set 是活的产物，半是 spec 半是训练数据，每个 prompt 都是塑造 agent 行为的样本。"

人人都是产品经理上有一篇被反复转的文章叫《PRD 退场倒计时》——里面 PM 吐槽：「PRD 中途变更或二期迭代时，**上下文丢失、历史业务约束被覆盖、引发全局格式崩塌**。」

不是 PM 写得不好，是 PRD 这个工具不够用了。

**写过 AI PRD 的 PM 通常会卡在 3 个章节**：

- **评估集（Eval）章节**——传统 PRD 没这一章，PM 不知道怎么定 case 数、通过线、谁来标
- **Bad Case 分级章节**——4 级降级策略要在 PRD 阶段就定，但没踩过坑就想象不出来
- **成本预算章节**——给 CFO 看的 token 账本，PM 一直依赖工程师而没自己算过

3 章卡其中 1 章，PRD 就写不下去。第 11-15 篇会把这 3 章逐个拆开讲。

## 崩塌点三：技术评审被反问"评估集呢"，答不上来

PRD 写完了，硬着头皮拿去评审。

工程师看完，问一句：

> 「你的评估集呢？」

PM 愣住。

传统 PM 不需要带评估集——带验收标准，QA 写测试用例，评审通过开发。

AI PM 必须带评估集。**没有评估集，工程师没法定义"做完了"是什么意思**。

Hamel Husain 和 Shreya Shankar 在 Lenny Newsletter 2025 年的核心观点是：

> "**Writing good evals is becoming the defining skill for AI PMs.**"

他俩教过 2000+ PM 和工程师，包括 OpenAI 和 Anthropic 团队。课上反复出现的崩塌瞬间就是——**PM 拿着 demo 走进评审，被工程师反问"你的 golden set 在哪、判分标准是什么"**。

PM 没准备过这一题。

不是 PM 偷懒。是 **「评估集」这件事在传统 PM 工作流里根本不存在**。SQL 不会还能当 PM，evals 不做就当不了 AI PM——这就是 Hamel 说的「**新时代的 analytics**」。

**做过 8 年质量工程的视角**——评估集其实不是新东西。任何传统 ML 项目（欺诈识别、流失预测、销量预测）都把 evals 作为基础设施。LLM 圈把这事丢了——只是因为 prompt engineering 看起来太"轻"，让人误以为可以跳过评估。**结果就是上线即翻车**。

## 崩塌点四：老板问"这功能成本多少"，token 账算不出来

PRD 评审通过了，开发开始了。这时老板找过来：

> 「这个功能上线之后，每个月烧多少钱？」

PM 不会算。

不是不会数学。是**根本不知道一次对话会触发多少 token、retry 多少次、上下文累积多深、缓存命中多少、output 比 input 贵多少倍**。

只能含糊其辞地说"应该不贵吧"。

老板半信半疑。

然后 PM 去问工程师。工程师说"按这个 prompt 长度算，一次对话大概 1500 token，按 GPT-5 输出价 $30/M……"

PM 听不懂。

更可怕的是 **Cursor 给所有 AI 产品 leader 演了一场公开课**：2025 年 6 月，Cursor 未预警切换到 credit-based billing，老用户从 "$20/月心智模型" 突然变成 $200-$400 的 overage，Hacker News 上有人晒 "$350 in a week"——相当于**月成本涨到 10-20 倍**。7 月 4 日 CEO 公开道歉退款。

这件事之后，每个 AI 产品 leader 的老板都开始问：「我们家会不会也这样？」

而大部分 PM 的答案是——「**我不知道**」。

**做 AI 产品咨询时反复看到的 3 个 token 账算错的坑**：

1. **算了 input 没算 output**——output token 比 input 贵 3-10 倍，老板看到的账单永远比预估的高 5 倍
2. **算了首次调用没算 retry**——失败 retry 1 次 = 成本翻倍，3 次 retry = 4 倍成本
3. **算了基础 prompt 没算上下文累积**——多轮对话的 input 不是线性增长，是**二次方膨胀**

3 个坑都踩过的 PM——通常已经被老板叫去过办公室谈话了。

（第 9 篇 Token 账本会把这 3 个坑 + 5 个常见漏算逐个拆开讲。）

## 崩塌点五：上线后 bad case 出现，不知道该修 prompt 还是改产品

硬着头皮上线了。

第三天，运营找你：「用户反馈，AI 在 X 场景下回答错了。」

打开后台，看了 5 个 case，开始修 prompt。

修完，上线。

第七天，运营又找你：「用户反馈，AI 在 Y 场景下回答错了。」

打开后台一看——这个 Y 场景，**上周明明跑得好好的**。

这是「**修了 A 坏了 B**」。

OpenAI 自家的 GPT-4o sycophancy 事件就是这个机制——为了让对话"更友好"，结果训练成附和有害决策（甚至附和停药）。OpenAI 自己的 postmortem 里承认：「**sycophancy 没有作为 blocking 项进入发版评估**。」

4 月 25 日推全 → 4 月 29 日全量回滚。**4 天**。

要不要回滚？要不要继续改？要不要承认这个 case 不是 prompt 能解决的，得改产品设计？

传统 PM 修 bug 像修水管——修好就是修好了。
AI PM 修 bad case 像调钢琴——**调一根弦其他弦都会响**。

工作流里没有"回归集"这个概念。脑子里没有"failure taxonomy"这个分类。只能拍脑袋。

**实际操作过 AI 项目的 PM 会发现**——同一个 bad case，改 prompt 改 3 轮没效果，根因压根可能不在 prompt 上：

- 是 **retrieval 漏了关键文档**（改 prompt 永远救不回来）
- 是 **业务流程本身不该让 AI 决策**（要改的是产品设计，不是 prompt）
- 是 **用户输入有 prompt injection**（要加 input filter，不是改 system prompt）

5 类失败模式（hallucination / refusal / drift / format breaking / safety violation），**每一类的修复路径完全不同**——但 90% 的 PM 第一反应都是"改 prompt"。

**质量工程视角看**——这是把"测试用例"和"分布修正"混为一谈。传统 bug 是 binary（出错 / 不出错），AI bad case 是 distributional（在哪个分布尾部出错）。**用修 bug 的思维去修分布——必然按下葫芦浮起瓢**。

（第 21 篇 Bad Case 管理会把 5 类失败模式 + 4 选 1 修复策略全部拆开讲。）

## 崩塌点六：复盘会上无法判断"是模型问题还是设计问题"

第一周复盘会。

数据不好看。留存低、转化低、用户投诉多。

工程师说：「模型还不行，等下个版本。」
设计师说：「用户路径有问题，得重新走查。」
PM 说：「……」

PM 说不出归因。

不是不专业。是 **AI 产品的失败有 4 类完全不同的归因路径**——hallucination（幻觉）、refusal（拒答）、drift（漂移）、format breaking（格式崩）。每一类的修复方法完全不同。Hamel 把这个叫 「**failure taxonomy**」。

但中文 PM 圈对这个概念几乎没有普及。

复盘会上，工程师把锅推给"用户路径"，设计师把锅推给"模型能力"，**最后没人动**。下一周数据继续差。再下一周，老板找过来——「为什么我们的 AI 不如 ChatGPT？」

PM 这才发现，**自己已经从「PM」变成了「被夹在中间不知道怎么办的人」**。

## 「不以为然」的真正代价：被甩开 10 倍速度

读到这里，6 个崩塌点应该已经清楚了。

但比 6 个崩塌点更扎心的是——**不以为然的 PM，不只是踩坑，是被新工作流以 5-10 倍速度甩开**。

具体什么意思？

| 工作场景 | 旧工作方式（不以为然的 PM）| 新工作方式（已转的 PM）|
|---------|------------------------|-------------------|
| 验证一个 AI 需求是否可行 | 写 PRD → 评审 → 排期 → RD 开发 → 等 demo（2-4 周）| 自己用 Claude Code 跑通验证（**2-3 小时**）|
| 写 PRD | 10 页 Word + 多次评审（3-5 天）| Product Note + Spec + 跑通原型（**1 天**）|
| 复现一个 bad case | 找 RD 排期 → 等 trace（3-7 天）| Claude Code 读 codebase 本地复现（**1 小时**）|
| 与工程师对齐能力边界 | 来回会议 + 反复 prompt 改（1-2 周）| 一起跑 eval 用数据说话（**1-2 小时**）|
| 给老板算 token 成本 | 找 RD 估算 → 来回核对（3-5 天）| 自己用 spreadsheet + Claude 算（**30 分钟**）|

**5 个场景平均下来——新工作方式快 10 倍**。

这意味着：
- **半年后**：用新工作方式的 PM 已经做了 10 个项目验证，「不以为然」的 PM 还在做第 1 个 PRD
- **一年后**：用新工作方式的 PM 已经形成产品判断力沉淀，「不以为然」的 PM 还在追"为什么 AI 不如 ChatGPT"
- **两年后**：用新工作方式的 PM 可能已经是某个 AI Native 团队的核心，「不以为然」的 PM 收到 HR 谈话邀请

**这不是技能差距，是认知差距**。

技能可以补——开个课、读本书、跟着教程跑一遍。
认知差距是隐性的——你不知道自己缺什么，因为你以为"不就是 PM 嘛"。

## 接下来 25 篇做什么

这一篇是序章。从下一篇开始，**逐个把旧工作方式换成新工作方式**：

- 第 2 篇先把范式切换说清楚——从「管功能」到「管能力」
- 第 3-6 篇修需求段（崩塌点一）
- 第 7-10 篇修方案段（崩塌点二、四）
- 第 11-15 篇修 PRD 段（崩塌点二、三）—— 这是专栏的硬通货段
- 第 16-19 篇修交付段（崩塌点三、五）—— **含 PM 用 Claude Code 的杀手锏**
- 第 20-22 篇修运营段（崩塌点五）
- 第 23-25 篇修协作段（崩塌点六）—— **含 AI Native 团队搭建（B 端漏斗核心）**
- 第 26 篇收尾——一个 PM 的 AI 时代成长路径

每一篇都按**双层结构**——上半篇是 **workflow**（你立刻能拿去用的方法、模板、判断框架），下半篇是 **engineering reality**（为什么这样做、底层的工程约束、踩坑案例）。

只看上半篇，能立刻在下周的工作里用上。
两层都看，会建立一套别处看不到的判断力。

## 写在最后：为什么写这 26 篇

如果你已经在 AI PM 圈看了几年内容，你可能注意到一件事——

业内不缺 AI PM 教程。Lenny / Marty Cagan / Sachin Rekhi / Hamel Husain / Aparna Chennapragada 都在写。极客时间、起点学院、三节课都有课。

那为什么还要再写一份？

**不是因为方法论独家**——这些方法论 Anthropic / OpenAI / Hamel 都讲过。

**是因为业内大多数 AI PM 内容只讲了"思维变了"那一层（崩塌点的根因 1），没系统讲"节奏变了 + 工具变了"那两层（根因 2 + 3）**。

8 年做软件质量工程 + 转 AI 产品看到的一个反共识——

> **AI 产品的下一个十年，赢家不是模型最强的，是质量门槛最高的；
> AI PM 的下一个十年，赢家不是知识最全的，是工作流最快的。**

26 篇就是想把「**质量工程视角看 AI 产品 + AI Native 工作流的实操方法**」这两件事完整地讲一遍——

- 不只是教你写 prompt
- 不只是教你选模型
- 不只是教你做 demo

是教你**整套从旧工作方式切换到新工作方式的具体步骤**——
- 怎么写新形态的 PRD（Product Note + Spec + 跑通原型）
- 怎么用 Claude Code 把验证周期从 2 周压到 2 小时
- 怎么搭 PM-Prompt-Eval 反馈环替代 PM-RD-QA 流水线
- 怎么用质量工程视角看 AI 产品的失败模式、概率分布、决策边界

——这是中外 PM 圈都还没系统讲过的位置。

如果你还在用「功能列表」评审 AI 产品，你和工程师永远聊不到一块去。
