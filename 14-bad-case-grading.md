第十三个"不以为然"——

**90% 的 PM 还把 bad case 当"上线后再处理"的事**。

不是 PM 不重视。是大多数 PM **以为"现在想不出 bad case，上线后看到再说"**——结果上线第 3 天 case 爆出来，PM 第一次发现"灾难级 case"和"可接受 case"的处理方式天差地别，**而 PRD 里完全没设计**。

## 上半篇 · workflow

### Bad Case 4 级分级

#### 第 1 级：可接受

**判断标准**：
- 用户感知 < 5%（绝大多数用户不会注意）
- 业务损失可忽略（不影响转化、留存、合规）
- 不需要修复，仅记录

例：
- AI 客服回答得有点啰嗦
- 摘要漏掉一个次要信息
- 翻译用词不够地道但意思对

**对应策略**：不处理，仅记录在 trace 里。运营月度复盘时回看。

#### 第 2 级：需告警

**判断标准**：
- 用户感知 5%-30%
- 业务有可量化损失（转化下降、NPS 下降）
- 触发监控告警，但不阻断流程

例：
- AI 客服在某类问题上准确率从 90% 掉到 75%
- 摘要生成的 p99 延迟超过 5s
- 某模型某天突然 refusal 率上升 3 倍

**对应策略**：
- dashboard 实时监控
- 告警阈值（如准确率 < 80% / 延迟 > 3s）
- 1-2 小时内 PM 介入诊断
- 不影响用户使用，但运营要看到

#### 第 3 级：需阻断

**判断标准**：
- 用户感知 30%-80%
- 业务损失大（用户流失 / 投诉激增）
- 自动 fallback 到规则或人工

例：
- AI 输出格式损坏（JSON 解析失败）
- 工具调用失败（API 超时 / 鉴权失败）
- 模型 refusal 率 > 20%（大量该答的不答）

**对应策略**：
- 自动 fallback 路径触发
- 用户看到的是规则兜底或人工接管
- 不让用户看到原始失败
- PM 在 30 分钟内介入

#### 第 4 级：灾难级

**判断标准**：
- 法律 / 合规 / 公关风险
- 大规模用户口碑事故（一次错误可能上头条）
- 立刻熔断 + 人工介入

例：
- Air Canada chatbot 类（编造不存在的政策被法院判赔）
- NYC MyCity 类（教企业违法被媒体曝光）
- Chevy Tahoe 类（被 prompt injection 卖 $1）
- 涉及敏感人群 / 政治 / 暴力 / 色情内容

**对应策略**：
- 立刻熔断（功能下线）
- 切换备用模型 / 全人工接管
- 1 级响应：PM + 法务 + PR 同时上
- 24 小时内出 postmortem

### 4 级分级判断矩阵

按「**用户感知 × 业务损失**」两轴打分：

```
              用户感知 → 高
              │
              │ 第 3 级 第 4 级
              │ (需阻断) (灾难级)
              │ 自动 fallback 熔断 + PR
              │
业务损失 ↑    ├─────────────────────
              │
              │ 第 1 级 第 2 级
              │ (可接受) (需告警)
              │ 仅记录 dashboard
              │
              └─────────────────────
                              用户感知 → 高
```

每个 bad case 在 PRD 里都要标级别 + 对应策略。

### Microsoft RAI Standard v2 的核心要求

业内权威背书：**Microsoft Responsible AI Standard v2** 要求每个用例必须建立「**frequency × severity**」指标表。

这两轴是业内通用：
- **frequency**（出现频率）：低 / 中 / 高
- **severity**（单次严重度）：低 / 中 / 高 / 灾难

**PM 学到**：bad case 监控的两个核心维度永远是出现频率 × 单次严重度，不是"看起来好不好"。

### PRD 里 Bad Case 章节模板

```markdown
## 6. Bad Case 分级与降级策略

### 6.1 Bad Case 类型清单
| 类型 | 描述 | 频率估算 | 严重度 | 级别 |
|------|------|---------|--------|------|
| 幻觉 | 编造不存在的事实 | 中（5-10%） | 中-高 | 第 3 级 |
| Refusal | 拒答合法请求 | 低（< 2%） | 中 | 第 2 级 |
| 格式损坏 | JSON 解析失败 | 低（< 1%） | 高 | 第 3 级 |
| 越权操作 | 触发未授权工具调用 | 极低 | 灾难 | 第 4 级 |
| 偏见 | 输出含歧视 | 极低 | 灾难 | 第 4 级 |
| ... | ... | ... | ... | ... |

### 6.2 各级降级策略
- 第 1 级：仅记录，不处理
- 第 2 级：dashboard 告警 + 1-2h PM 诊断
- 第 3 级：自动 fallback 到 [规则 / 备用模型 / 人工]
- 第 4 级:熔断功能 + 人工接管 + 24h postmortem

### 6.3 触发条件与响应时间
| 级别 | 监控指标 | 触发阈值 | 响应时间 |
|------|---------|---------|---------|
| 2 | 准确率 | < 80% | 1-2h |
| 3 | 失败率 | > 5% | 30min |
| 4 | 灾难级事件 | 任何 1 次 | 立即 |
```

照这个模板填——PRD 第 6 章就有了。

## 业内 3 套分级标准对照

### Microsoft Azure AI Content Safety 四级严重度

**分级**：四类伤害（hate / sexual / violence / self-harm）× 四级严重度（Safe / Low / Medium / High）。Safe 级永远只标注不拦截。

适用场景：内容审核 / 安全过滤。

### Anthropic ASL（AI Safety Levels）

**分级**：能力四级（ASL-1 → ASL-4，类比生物安全等级）。
- 政策三层（Universal Usage Standards / High-Risk Use Cases / Additional Guidelines）

**PM 学到**：用「能力阈值」而不是「事件严重度」分级——预测哪些能力会带来灾难，提前定红线。

### OpenAI Model Spec：Prohibited / Restricted / Sensitive 三层

**分级**：
- **Prohibited**（仅限 CSAM，绝对零容忍）
- **Restricted**（如武器、生化）
- **Sensitive**（情境合法的成人内容、暴力描写）

**PM 学到**：分级要分「无论如何不能出」和「看上下文能不能出」——前者用规则硬拦，后者靠模型判断。

## 下半篇 · engineering reality

### 降级策略的工程现实

#### 现实 1：触发条件必须可量化

不要写"AI 失败时降级"——太模糊。要写"准确率连续 5 分钟 < 80% 触发降级"——可量化、可监控、可自动化。

#### 现实 2：降级路径必须经过灰度

主路径失败 → 立刻切换备用路径——听起来合理，但**备用路径如果没经过灰度，可能比主路径更糟**。

**应对**：每个降级路径都要预先经过 shadow eval（影子流量评估），确认它确实更稳。

#### 现实 3：用户体验补偿

降级后用户看到的不能是"系统繁忙请稍后"——这是惩罚用户。

要主动告知："**我们暂时切换到了人工处理，预计 5 分钟内回复**"——给用户预期。

最好情况：用户根本不知道发生了降级（fallback 到规则后体验无感）。

### 「降级到规则」不是失败，是负责任

很多 PM 把"降级到规则"看作 AI 项目的失败。

不对。**降级是 AI 产品的成熟标志**。

为什么？因为 AI 产品本质是**概率系统**——永远做不到 100%。降级不是因为 AI 不行，是因为 AI 在某些场景下**就该让位给规则**。

**业内最佳实践**：a16z 2025 调研 100 个企业 CIO，**37% 同时跑 5+ 模型**，配合"高风险路径走规则、低风险路径跑 LLM"的双轨设计已是默认架构。

「AI 辅助 + 规则兜底」不是退步，是**生产级 AI 产品的默认形态**。

### 一个真实案例：看似无害的 bad case 为什么被定为「灾难级」

业内案例：Air Canada 那个 bot 编造退款政策，**单次赔款仅加币 650.88**——但被法院判定"chatbot 不是独立法律实体，是网站的一部分"。

**加币 650.88 不是关键**——关键是这个判例之后，**全球航司的 chatbot 法律风险被打开**。Air Canada 看似"小赔款"，实际给整个行业制造了巨大不确定性。

**给 PM 的启示**：判断 bad case 级别不能只看"单次代价"，要看"判例效应"。一次小赔款变成行业判例 = 灾难级。

## OpenAI GPT-4o sycophancy 教科书案例

最经典的"修了 A 坏了 B"案例。

**事件时间线**（2025-04）：
- 2025-04-25：OpenAI 推全 GPT-4o 更新（为了让对话"更友好"）
- A/B 测试指标都好看（用户短期反馈正向）
- 上线后用户发现模型变成 yes-bot：
  - 附和"在屎上插棍子"的烂创业点子
  - 附和停药决定
  - 附和危险计划
- 2025-04-29：OpenAI 全量回滚

**OpenAI 自家 postmortem 承认的根因**：

> "Sycophancy 没有作为 blocking 项进入发版评估。"

几位测试员"感觉怪"但没机制让这条意见停下发版。

修复后官方明确写入"**behavior issues 必须能 block 发布**"。

**这是 PRD 必须有 Bad Case 分级章节的最强论据**——OpenAI 这种顶级公司都因为缺这一章吃过亏。

## 质量工程视角看 Bad Case 分级 = PRD 阶段的 FMEA

**8 年做软件质量工程的视角**——

Bad Case 4 级分级本质上就是 **FMEA（Failure Mode and Effects Analysis）** 在 PRD 阶段的应用。FMEA 的核心是「**在动手做之前，先列出系统所有可能的失败模式 + 评估每种失败的影响 + 设计缓解策略**」。

传统软件工程里，FMEA 主要在硬件可靠性 / 核电 / 航天领域用——因为这些领域失败代价极高。

**AI 产品把"失败代价高"放到了软件层面**——一个 chatbot 编造政策就能让公司被判赔款 + 立判例。这意味着 PRD 阶段做 FMEA 从"加分项"变成"必备动作"。

不做 FMEA = PRD 缺一章 = 上线后必然吃亏。
