# PRD：4 增 2 删 完整模板 v1.0

> 服务于第 11 篇《AI 时代 PRD 骨架》配套资源 ①
> 基于「4 增 2 删」原创框架 —— AI 产品 PRD 必须新增 4 节、删除 2 节
> 配套：第 12 篇 Eval 集 / 第 14 篇 Bad Case 分级 / 第 15 篇 成本预算 / 第 19 篇 GO-NOGO

---

## 一、为什么是「4 增 2 删」

> **Aparna Chennapragada（Microsoft CPO）** 在 Lenny's Podcast 直言：
>
> "**Prompt sets are the new PRDs**. Prompt set 是活的产物——半是 spec，半是训练数据。传统 PRD 为程序员而写、提前锁死需求再交付，但 AI 产品本质是非确定性的，需求文档无法被一次性翻译成产品。"
>
> **Cat Wu（Anthropic Claude Code Head of Product）** 在同一档播客补充：
>
> "**Prototypes over PRDs**. 能用 Claude Code 做出原型的就不写 spec。但'变薄'前提是有 `context.md` 做支撑 + 团队有 AI 编码闭环——普通团队直接抄'3 段产品笔记'只是偷懒。"

这两句话是 2026 年 AI PM 圈最广为流传的口号。但它们各自只解决了**一半问题**：

- Aparna 说"PRD 该变薄"——对，但**变薄到哪去**？没说清。
- Cat 说"原型优于 PRD"——对，但**原型撞墙时回退到什么文档**？没说清。

**「4 增 2 删」是中间道路**——传统 PRD 模板已被 20 年瀑布模式验证有效，不必推倒重来；但确定性系统的假设在 AI 产品里彻底失效，必须新增 4 个章节兜住"非确定性"，同时砍掉 2 个传统 PRD 里**写了也是浪费**的章节。

**核心论点**：
- **加的 4 块** = 把「质量工程基本动作」从工程师内部文档**提升到 PRD 写明** —— 评估集 / Bad Case / 降级 / 成本
- **砍的 2 块** = 把「Lovable / v0 / Cursor / Spec Kit 比 PM 画得快」的章节**让给 AI 工具** —— 详细 UI / 完整流程图

**90% 的 PM 还在用传统模板加几个 AI 章节**，结果工程师按这份 PRD 做出来的东西**必然在生产环境崩** —— 因为传统 PRD 默认是确定性系统，与 AI 的概率系统本质冲突。

## 三、模板正文 1-10 节完整展开

### 第 1 节：产品背景 + 用户故事

**填写说明**：写"为什么现在做"，不要写"为什么 AI 是好东西"

**模板**：
```markdown
## 1.1 产品背景

- 业务现状：[现在用户在 X 场景靠 Y 解决，但 Y 有 Z 问题]
- 现在做的理由：[最近发生 W 变化（成本/技术/合规），机会窗口在 N 个月内]
- 不做的代价：[竞品做了 / 用户流失到 X / 监管要求]

## 1.2 用户故事（每个核心角色 1 条）

- As a [具体角色：电商运营 / 金融客服 / 法务专员，不写"用户"]
- I want [具体行为：自动提取 PDF 关键条款，不写"智能识别"]
- So that [具体价值：单合同审核时间从 3h 降到 30min，不写"提效"]
```

**禁止清单**：
- ❌ "As a 用户 I want a 智能助手"
- ❌ "让公司全面 AI 化"
- ❌ "提升 X% 智能化水平"

**验收标准**：
- ✓ 每条用户故事的「角色」是岗位名
- ✓ 每条「行为」能拍照拍下来（不是"提升"/"优化"）
- ✓ 每条「价值」有可对比的基线数字

### 第 2 节：核心场景 + 用户旅程

**填写说明**：每个核心场景必须含 4 项结构化字段

**模板**：
```markdown
## 场景 [N]：[一句话场景名]

| 字段 | 描述 |
|---|---|
| 触发条件 | 用户点击 X / 收到 Y 邮件 / 每日 8:00 定时 |
| 用户输入 | 自由文本 ≤ 2000 字 / PDF ≤ 20 页 / 结构化表单 |
| 期望输出 | JSON {key1: ..., key2: ...} 或 自然语言 ≤ 500 字 |
| 输出验收点 | 1) key1 必填 2) 总分 0-100 3) 列出 top 3 reasons |
| 失败兜底 | LLM 异常 → 返回兜底文案"系统繁忙，请人工处理"+ 转工单 |
| 频率/规模 | 估算每日 10 次 / 1000 次 / 10W 次 |
```

**验收标准**：每个场景 4 项都填了（缺 1 项不通过）+ 失败兜底是**具体动作**而非"加监控"

### 第 3 节：功能列表 + 优先级

**填写说明**：每个功能必须标 3 个属性

**模板表格**：
```markdown
| # | 功能 | 优先级 | AI 介入度 | 验收口径 | 备注 |
|---|---|---|---|---|---|
| F1 | [一句话功能名] | P0 | AI 自治 | Eval 集核心 case ≥ 85% | |
| F2 | | P0 | AI 辅助 | 人审通过率 ≥ 95% | 高风险走人工兜底 |
| F3 | | P1 | 规则 | 单元测试覆盖率 100% | 不上 AI |
```

**关键判断**：
- 「AI 自治」功能必须有对应 Eval 集（第 8 节）
- 「AI 辅助」必须写清"人审在哪个环节" + "人审 SLA"
- 「规则」功能不要包装成 AI

### 第 4 节：非功能需求（NFR）

**填写说明**：传统 NFR 5 维 + 3 个 AI 特有维度

**模板**：
```markdown
## 4.1 性能
- 单次调用 P50 ≤ X 秒 / P99 ≤ X 秒
- 并发：QPS 峰值 X / 平均 X
- 超时设定：模型调用 ≤ X 秒；总链路 ≤ X 秒

## 4.2 安全
- 已通过第 04 篇《伪 AI 需求 6 问》Q4 + Q5 审查
- Prompt injection 防护：[白名单 / 输入 sanitize / 输出过滤]
- 数据脱敏：用户 PII 进入 prompt 前必须脱敏

## 4.3 合规
- 已通过 Microsoft RAI v2 frequency × severity 矩阵评估（见第 9 节）
- 涉及的法规：[GDPR / PIPL / Illinois AI Bill / Texas TRAIGA]
- 用户告知义务：UI 标"AI 生成"水印 / 用户可关闭

## 4.4 可用性
- 服务整体 SLA 99.5%

## 4.5 可观测
- 必备指标：每次调用的 input/output/latency/cost/user_id（trace 级别）
- 用户反馈：点赞 / 踩 / 重新提问的埋点
- 绕过 AI 的行为埋点

## 4.6 AI 特有 1：模型选择 + 备用模型
- 主模型：[GPT-5 / Claude Sonnet 4.7 / Qwen-Max]
- 备用模型：[同档次模型]
- 切换条件：主模型 5xx > 1% 持续 5min → 自动切

## 4.7 AI 特有 2：回退路径（cascading fallback）
- 第 1 级：主模型
- 第 2 级：备用模型
- 第 3 级：规则兜底
- 第 4 级：人工 queue

## 4.8 AI 特有 3：可解释性
- 用户能看到的：[输出 + 引用源 / 输出 + 置信度 / 仅输出]
- 内部能查到的：每次调用的 prompt 全文 + 模型版本（审计保留 N 月）
```

### 第 6 节：发布计划（含 AI 灰度 4 阶段）

**模板**：
```markdown
| 阶段 | 用户比例 | 时长 | 核心监控指标 | GO 条件 | NOGO 条件 |
|---|---|---|---|---|---|
| Shadow（影子） | 0%（对照旧逻辑跑） | 3-7 天 | 输出一致性 / 成本 / 延迟 | 与基线偏差 ≤ X% | 偏差超阈值 |
| Canary 1% | 1% 流量 | 3-5 天 | Bad Case 率 / 用户反馈 | Eval 通过率 ≥ 85% + 用户反馈无 L3+ | 任一 L4 / Bad Case > 阈值 |
| Canary 10% | 10% 流量 | 5-7 天 | 成本 MAU / Bad Case 分布 | 成本 ≤ 预算 80% + Bad Case 收敛 | 成本超 100% / L3+ 集中 |
| 全量 | 100% | - | 全部指标 | 前 3 阶段全 GO | 任意阶段 NOGO 立刻回滚 |
```

**关键纪律**：
- 任一阶段 NOGO = **立即回滚到上阶段**（不是"再观察一下"）
- 阶段切换必须有 PM/工程/数据三方签字
- 全量后**前 2 周保留一键回滚开关**

### ➕ 第 7 节：能力边界（AI 产品必填）

**理论依据**：参照 OpenAI Model Spec 三层结构

| 层级 | 含义 | 谁能改 | PRD 写法 |
|---|---|---|---|
| **Rules**（强制） | 绝不做的事 | 不可被覆盖 | "永远不输出 X" |
| **Objectives**（目标） | 应该做到的事 | 可被特例覆盖 | "尽可能 Y" |
| **Defaults**（默认） | 默认行为 | 可被用户/开发者改 | "默认 Z，用户可关" |

**模板**：
```markdown
## 7.1 我们做什么（Capability — 3-5 项具体能力）
1. [能力名]：[输入形态 → 输出形态 + 准确率目标]
   例：「合同关键条款提取」：PDF → JSON {甲乙方/金额/期限/争议条款}，准确率目标 90%
2. ...

## 7.2 我们不做什么（Rules — 5-10 项绝不做的事）⭐ 比 7.1 长
1. 永远不做：法律意见 / 处方建议 / 投资决策 — 涉及高合规风险
2. 永远不做：自动签署合同 / 自动付款
3. 永远不做：基于用户身份的差异化定价 — 涉及反歧视法
4. 永远不做：未经用户同意将对话用于训练
5. 永远不做：跨用户检索他人 PII

## 7.3 模糊地带（Defaults — 默认行为 + 可改）
| 用户问 | 我们的默认动作 | 用户可改吗 |
|---|---|---|
| 涉及竞品对比 | 拒答 + "建议自行调研" | 否（Rule） |
| 涉及主观评价 | 给中性描述不给评价 | 否 |
| 涉及代码安全建议 | 给建议 + 标注"非安全审计" | 是（开发者可禁用） |

## 7.4 升级路径（Anthropic ASL 风格）
- 当前 Capability Level：CL-2（人工兜底场景）
- 升级到 CL-3（AI 自治）的前置条件：连续 3 个月 L3+ Bad Case = 0 + NPS ≥ X
```

**验收标准**：
- ✓ 7.2 行数 > 7.1 行数（不做的比做的多）
- ✓ 每条 Rule 是"永远不"（不是"尽量不"）
- ✓ 模糊地带至少 5 类，每类有明确动作

**真实反例**（PRD 第 7 节缺失的代价）：
- **Chevy 经销商 chatbot（2023-11）**：用户让 bot "agree with anything"，bot 同意以 $1 卖 $80K 的 Tahoe。**PRD 缺 Rule**：「涉及价格/合同必须人工确认」
- **NYC MyCity（2024-03）**：bot 告诉商家"可以拿员工小费"。**PRD 缺 Rule**：「涉及法律合规话题必须降级到人工」
- **Air Canada（2024-02）**：bot 编造丧亲票政策。**PRD 缺 Rule**：「涉及公司政策必须 RAG grounding 引用真实文档」

### ➕ 第 8 节：Eval 集 + 上线 GO/NOGO 5 维 checklist

**理论依据**：参照 **Scrum.org《Definition of Done for AI Agents》**：

> "**AI 是概率性的——同一个 prompt 跑 100 次可能 95 次对 5 次幻觉。如果你的 DoD 还是 Pass/Fail 二值检查，你不是在测试 agent，你在赌博。**"

**模板**：

```markdown
## 8.1 Eval 集设计（详见第 12 篇深入）
| 类型 | Case 数 | 来源 | 标注方 | 跑评频率 |
|---|---|---|---|---|
| 核心场景 cases | ≥ 50 | 真实历史用户输入 | PM + 业务 | 每次 commit |
| 长尾场景 cases | ≥ 30 | 客服工单 / 用户反馈 | 运营 | 每次 release |
| 对抗场景 cases | ≥ 20 | Red team 设计 | 安全 + PM | 每周回归 |
| 合计 | ≥ 100 | | | |

注：Anthropic 官方建议「先用 20-50 个真实失败 case 起步」。
先做 5000 条 case 再上线的团队，往往一个 case 都用不上。

## 8.2 上线 GO/NOGO 5 维 checklist（参照 Miqdad Jaffer AI PRD Template）

| 维度 | 阈值 | 实测 | GO/NOGO |
|---|---|---|---|
| 1. 核心 case 通过率 | ≥ 85% | __% | |
| 2. 性能 P99 | ≤ X 秒 | __s | |
| 3. 单次成本 | ≤ X 元 / 单 MAU ≤ X 元 | __ | |
| 4. 对抗 red team | 已跑 + 0 个 L4 case | __ | |
| 5. 历史回归 | 衰退 ≤ 5% | __% | |

⚠️ **任一维度不达标 = NOGO**，没有商量余地。

## 8.3 持续 Eval 节奏
- 每次 commit → 跑 8.1 第 1 行（核心 50）
- 每次 release → 跑 8.1 全表 100+
- 每周 → 跑 8.1 + 历史回归集
- 每月 → 重新审视 Eval 集是否需要扩充新 case
```

### ➕ 第 9 节：Bad Case 分级 + 降级策略

**理论依据**：参照 **Microsoft Responsible AI Standard v2** 的 frequency × severity 2D 矩阵思路。

**模板**：

```markdown
## 9.1 Bad Case 4 级分级（详见第 14 篇深入）

| 级别 | 定义 | 频率必须 < | 严重度 | 触发动作 |
|---|---|---|---|---|
| L1 提示 | 输出不完整 / 风格不达预期 | 5% | 用户可重试 | 监控 + 周报 |
| L2 告警 | 事实小错 / 引用错误 | 1% | 影响小批量 | dashboard 告警 + PM review |
| L3 阻断 | 业务流程错误 / 大批量影响 | 0.1% | 影响业务 | 自动 fallback + 立即修复 |
| L4 灾难 | PR 事故 / 合规违规 / 财产损失 | 0（绝对 0 容忍） | 公司层面 | 立即降级 + 复盘 + 法务介入 |

⚠️ L4 频率必须是 **0**——不是 0.01%，是 **绝对 0**。

## 9.2 frequency × severity 矩阵（Microsoft RAI 风格）

           严重度 →
频率 ↓     | 低 (Cosmetic) | 中 (Business) | 高 (Reputation) | 极高 (Legal) |
高 (>5%)   |     L1        |     L2        |   ⚠️ 立即停服    |  ⚠️ 立即停服  |
中 (1-5%)  |     L1        |     L2        |     L3          |    L4         |
低 (<1%)   |     监控       |     L1        |     L3          |    L4         |
极低(<0.1%)|     -         |     监控       |     L2          |    L4         |

注：进入 ⚠️ 区 = 不上线。已上线进入 ⚠️ 区 = 立即停服。

## 9.3 降级路径（详见第 22 篇 4 场景）

| 触发条件 | 降级到 | 恢复条件 |
|---|---|---|
| 默认（高风险场景） | AI 辅助 + 人工兜底 | 不恢复（设计如此） |
| 默认（强合规场景） | 规则 + AI 仅辅助提示 | 不恢复（设计如此） |
| P99 > 阈值 | 规则 + 缓存 | 模型恢复 + 观察 30min |
| 主模型 5xx > 1% × 5min | 备用模型 | 主模型恢复 + 1h 观察 |
| 长尾 Bad Case > 20% | 头部用 AI + 长尾转人工 | 长尾失败率回到 < 5% |
| 月成本 > 预算 100% | 见 10.3 Hard Cap | 成本回到 < 80% |
| 任一 L4 case | 立即停服 + 全量回滚 | 复盘 + 修复 + 重新走 8.2 5 维 |
```

```markdown
|---|---|

## 10.3 Hard Cap 决策树

月成本累计监控（按日累加）
   │
   ├─ < 月预算 80% → 正常运行
   │
   ├─ ≥ 80% → 🟨 告警 → 邮件通知 PM + CTO + CFO
   │
   ├─ ≥ 100% → 🟧 自动降级 → 主模型 → 便宜模型 + 关闭非核心场景的 AI 调用
```

### ➖ 删除 1：详细 UI 规格

**不再写**：
- ❌ 每个按钮的颜色 / 大小 / 位置
- ❌ 每个页面的栅格 / 间距 / 字体
- ❌ 完整的高保真原型 / Figma 标注稿

**改为**（轻量 1 段话 + 关键交互目的）：
```markdown
## UI 设计意图（轻量版）

### 设计风格（1 段话）
[简洁现代 / 类 Notion 风格 / 暗色优先 / 移动端优先 ...]

### 关键交互意图（不是「样子」，是「目的」）
- 主输入区：让用户感受到"像跟同事说话"，不是"在填表"
- 结果展示：默认折叠 + 一键展开，避免信息过载
- 错误状态：明确告诉用户「错在哪 + 下一步怎么办」

### 关键页面（列名字 + 1 句话目的，不画 mockup）
- 主对话页：目的是减少用户思考"该怎么问"
- 历史记录页：目的是让用户能找回上次对话
- 反馈页：目的是给用户安全感（错了能投诉）
```

**理由**：Lovable / v0 / Cursor / Claude Artifacts 比 PM 画 Figma 快 10 倍

### ➖ 删除 2：完整流程图

**不再写**：
- ❌ 用 Visio / drawio 画的详细 flow chart
- ❌ 每个分支的状态机图

**改为**（关键决策点 + 分支条件）：
```markdown
## 流程关键决策点（让 AI 推导完整流程）

### 决策点 1：[用户输入合法性]
- 输入 ≤ 2000 字 + 无敏感词 → 进入决策点 2
- 输入 > 2000 字 → 提示截断 / 拒绝
- 输入含敏感词 → 拒答 + 引导

### 决策点 2：[意图识别]
- 意图 ∈ 白名单 → 进入对应 handler
- 意图 ∉ 白名单 → 拒答模板

### 决策点 3：[输出过滤]
- 输出含 PII → mask + 警告
- 输出含违规词 → 重新生成（最多 3 次）
- 3 次仍违规 → 兜底文案 + 转人工
```

**理由**（参照 **GitHub Spec Kit** Spec-Driven Development）：
> "**Specifications become executable, directly generating working implementations rather than just guiding them.**"

PM 只需描述关键决策点 + 分支条件，让 Claude Code / Cursor / Spec Kit 推导出完整流程

## 六、设计依据 & 批判性自评 v1.0 ⭐

### 6.1 这套模板的 5 个全球 tier-1 参照

| 框架 | 我借鉴了什么 | 我没采用什么 | 原因 |
|---|---|---|---|

| **OpenAI Model Spec** | Objectives / Rules / Defaults 3 层 → 第 7 节能力边界直接对应 | Model Spec 完整 50+ 页针对模型行为 | Model Spec 是模型行为规范，PRD 是产品决策——只借用 3 层结构思想 |

### 6.2 为什么选「4 增 2 删」而不是其他替代方案

| 替代方案 | 优势 | 缺点 | 拒绝原因 |
|---|---|---|---|
| **Mike Krieger Product Note 3-4 段** | Anthropic 内部已验证 + 极简 | 前提是有 context.md + orchestrator + Editor-in-Chief 闭环 | **中国团队没这闭环——直接抄等于"PM 偷懒"** |
| **Aparna Prompt Sets as PRDs** | 概念前沿 | "prompt 集 = spec"对工具链要求极高 | **目前国内 90% 团队还在写传统 PRD，激进迁移会失败** |
| **Miqdad 完整 9 节 AI PRD Template** | 大厂级完整度 | 9 大 section 中小团队跑不动 | **太重，5 人小团队填一份要 1 周** |
| **完全推倒传统 PRD + 全新 AI PRD 模板** | 哲学上一致 | 团队学习成本巨大 + 工程师不接受 | **离地三尺—— PRD 是 PM 和工程师的契约** |
| **「4 增 2 删」**（本版） | 在传统 PRD 基础上做增量改造 + 每节都有可执行模板 + 兼容渐进迁移 | 没真 PM 测试过 v1.0 | **✅ 选这个** |

### 6.4 适用 / 不适用边界

- ✅ **适用**：增量项目（已有功能加 AI）+ 全新项目
- ✅ **适用**：5-50 人规模团队（再小用 Krieger 3 段产品笔记，再大用 Miqdad 完整 9 节）
