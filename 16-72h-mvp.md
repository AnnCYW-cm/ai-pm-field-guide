# 16 · 72 小时 MVP 方法论——从 PRD 到可演示 Demo

> 第一季《AI 时代 PM 工作流重构》· 第 16 篇 · 第四部分 交付
> 预估字数：5000

---

第十五个"不以为然"——

**90% 的 PM 还在做"完整版小一点"的 MVP**。

不是 PM 不知道 MVP 概念。是大多数 PM **以为"MVP 就是把完整产品砍掉 80%"**——结果做了 1 个月还没上线，每个能力都做了点皮毛，**演示给老板时老板看不懂在干嘛**。

读完这一篇你会知道——**AI 产品的 MVP 不是"小一点的产品"，是"能演示能力闭环的最小单元"。72 小时是上限，不是目标**。

## 上半篇 · workflow

### 72 小时拆解

#### Day 1：PRD + 原型（8 小时）

```
2 小时：写「Product Note」（Anthropic 风格，3-4 段）
  - 用户意图（1 段）
  - 期望结果（1 段）
  - 具体指标（1-2 段）

2 小时：写 Eval 集（20-50 case）
  - 3 个评估维度
  - 通过线

2 小时：画能力流图
  - 3-5 个核心能力
  - 依赖关系 + 降级路径

2 小时：选模型 + 跑能力探针
  - 4 维度对比（成本 / 延迟 / 效果 / 合规）
  - 跑 5-10 个真实 case 验证可行性
```

**Day 1 结束产出**：能跑的 PRD 骨架 + Eval 集 + 模型选型决策。

#### Day 2：模型集成 + 核心能力（16 小时）

```
4 小时：搭基础脚手架（Lovable / Bolt / v0 / Claude Code 选一个）
  - 用 vibe coding 工具（详见下一篇第 17 篇）
  - 不要从零写

8 小时：集成核心能力
  - 1-2 个核心能力跑通（不是全部）
  - 简陋 UI（不要美化）
  - 不接生产数据库 / auth / 支付

4 小时：跑 Eval 集
  - 看通过率
  - 标 bad case
  - 决定要不要继续
```

**Day 2 结束产出**：能跑通核心能力闭环的原型。

#### Day 3：内测 + 演示（8 小时）

```
2 小时：找 3-5 个目标用户试用
  - 观察行为不收集愿望
  - 记录"卡在哪里"
  - 找出 happy path 之外的 case

3 小时：修 critical bug
  - 只修 4 级灾难级 + 3 级阻断级
  - 不修 1-2 级（演示用，不影响）

3 小时：录演示
  - Loom 录 3-5 分钟视频
  - 配 1 页 Notion ROI 推演
  - 不需要正式 deck
```

**Day 3 结束产出**：可演示的 demo + 录屏 + 决策文档。

### 演示对象决定 MVP 形态

3 种 MVP 形态长得不一样：

#### 形态 1：给老板演示
- **目标**：拿到立项 / 继续投入决策
- **形态**：Loom 录屏 + 1 页 Notion ROI 推演
- **不要做**：花哨 UI / 完整产品
- **重点**：让老板看懂"为什么这是 AI 能干的"

#### 形态 2：给客户演示
- **目标**：收 LOI / 付费意向
- **形态**：Lovable / Bolt 部署的 demo 链接 + waitlist
- **不要做**：接真实数据 / 用户认证
- **重点**：让客户能点 + 能感受核心能力

#### 形态 3：给团队对齐用
- **目标**：建立共同认知 / 解决评审分歧
- **形态**：v0 出 UI + 一个最长 happy path 走通
- **不要做**：完整流程 / 多角色权限
- **重点**：让团队"看见同一个东西"再讨论

### 真实成功案例

#### Lovable 自己
2023 年 Stockholm 创立——**2 个月做到 $10M ARR**，8 个月后 $200M @ $1.8B 估值——欧洲增长最快的初创纪录。

#### thoughtbot 工程师 weekend MVP
单人在一个周末用 Lovable 做出 ShareCheck（链接可信度检查）和 Hablayaya（西语练习）两个**真实可用的产品**。

#### AI Agents Benchmark
用 Lovable / Bolt / Replit / v0 / Base44 / Totalum 各自做出 vibeCrm 多租户 SaaS——**多个工具 < 1 天就能产出可演示 demo**。

### 各工具 MVP 时长对比

| 工具 | 时长 / 表现 |
|------|-----------|
| **Lovable** | 典型 web demo **< 10 分钟**可跑（含 Supabase）|
| **Bolt.new** | 选完 stack 后 8-10 分钟出脚手架 |
| **Replit Agent** | 复杂应用 ~18 分钟，但稳定性差 |
| **v0** | UI/前端最强，后端弱 |
| **Base44** | 开箱即用功能最全 |

## 下半篇 · engineering reality

### MVP 不该做的 5 件事

#### 1. 完整产品
MVP 不是产品，是**能力验证**。

完整产品需要：auth / 支付 / 邮件 / 推送 / 客服 / 合规——这些 72 小时做不完。

**应对**：**MVP 不接真实生产**。auth 用 hardcoded 用户、支付假数据、邮件 console.log、客服省略。

#### 2. 高质量 UI
UI 漂亮会让 PM 和工程师以为"差不多可以上线了"——**这是危险的错觉**。

MVP 是用来"看核心能力跑不跑通"——UI 应该**故意丑**，让所有人聚焦核心能力。

**应对**：用 Tailwind 默认样式 / shadcn 组件 / 系统字体。不要花时间做 mockup。

#### 3. 完整 eval
MVP 阶段做 vibe check 就够（自己看 10-20 个 case）。

正式 eval（50-200 case + 多人标注 + 通过线）留到上线前。

**应对**：写一个简版 eval（10 case），跑通即可。正式 eval 第 3 周再补。

#### 4. 性能优化
MVP 阶段不要优化延迟、不要优化 token 成本、不要优化并发。

那些都是上线前的事。MVP 阶段你要验证"**核心能力跑得通**"，不是"**核心能力跑得快**"。

**应对**：用最贵的模型（如 Opus 4.7）跑通，确认效果。上线前再降到 Haiku 类便宜模型。

#### 5. 完整错误处理
MVP 阶段错误就让它崩——你自己看 trace 修。

完整错误处理（try/catch / fallback / retry）留到上线前。

**应对**：MVP 只写 happy path。99% 的边界情况和错误不处理。

### 「可演示」vs「可上线」的鸿沟

MVP 不能直接变产品。**两者之间至少差一倍工作量**。

| 维度 | 可演示 MVP | 可上线产品 |
|------|----------|----------|
| 用户数 | 5-10 个内测 | 1000+ 真实用户 |
| 并发 | 1 个 | 100-1000 |
| 错误处理 | 没有 | 完整 fallback |
| 数据 | mock 数据 | 生产数据库 |
| auth | 假登录 | 真实鉴权 |
| 合规 | 不考虑 | GDPR / 数据隔离 |
| 监控 | 没有 | Dashboard + 告警 |
| 灰度 | 没有 | 多轴灰度 + shadow eval |
| eval | vibe check | 完整 eval + CI |
| 成本控制 | 不考虑 | hard cap + 熔断 |

**从 MVP 到上线产品至少 4-8 周**——不是 72 小时再翻一倍。

PM 必须清楚这一点。不要因为 MVP 做出来就跟老板说"下周可以上线"——**那是吹牛**。

### 72 小时的代价管理

**超过 72 小时还没出 MVP 的需求，基本是没收敛清楚的需求**。

为什么？因为 72 小时是 vibe coding 工具的临界点：
- < 24 小时：单一能力验证
- 24-72 小时：核心能力闭环
- > 72 小时：你在做的不是 MVP，是产品

如果你超过 72 小时——回去重新看需求：
- 是不是需求太大了？拆成 2-3 个 MVP
- 是不是路径选错了？Prompt → Agent → Fine-tune
- 是不是模型选错了？跑能力探针重选
- 是不是工具选错了？换 vibe coding 工具

**不要硬撑**。硬撑出来的 MVP 99% 跑偏。

## 质量工程视角看 MVP = 最小可证伪单元

**8 年做软件质量工程的视角**——

MVP 在质量工程里有一个更精准的名字——「**最小可证伪单元**」（Minimum Falsifiable Unit）。

科学方法的核心是「**提出假设 → 设计实验 → 证伪**」。MVP 就是"设计实验"那一步。

**实验设计的关键不是"做出完整产品"，是"快速证伪关键假设"**——
- 假设 1：用户会用 AI 解决这个问题（要证伪：用户给 demo 用了不用？）
- 假设 2：模型能稳定输出可接受结果（要证伪：eval 通过线达不达标？）
- 假设 3：商业模式能闭环（要证伪：单用户成本算不算得清？）

3 个假设任何一个被证伪——**这个产品就不该立项**。

72 小时 = 3 个假设的最短证伪周期。超过 72 小时 = 你在做的不是证伪实验，是做产品。
