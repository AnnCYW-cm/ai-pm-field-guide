## 上半篇 · workflow

### 成本章节 3 层结构

#### 第 1 层：单位成本（per-request）

```
单次调用成本 =
  (input_tokens × 输入单价) +
  (output_tokens × 输出单价) +
  (cache_read_tokens × cache 单价) +
  (其他附加成本：embedding / 工具调用 / 第三方 API)
```

⚠️ **CFO 第一次看到都会被打脸的点**：**output 比 input 贵 3-10 倍**。

典型 chatbot 实算：
- input：1M token × $0.15 = $0.15
- output：2M token × $0.60 = $1.20
- 合计：**$1.35/M tokens**——比官网"$0.15"贵 9 倍

### 第 2 层：规模成本（per-MAU / per-month）

**不要给 CFO 看 token，给他看 RMB / 月活用户**。

```
月度 LLM 成本 =
  单次调用成本 × 月均调用次数/用户 × MAU × (1 + retry 系数) × 头部 1% spike 系数
```

举个例子。客服 chatbot：
- 单次调用成本 ¥0.05
- 用户月均 30 次对话
- MAU 10 万
- retry 系数 1.3
- 头部 1% spike 系数 1.5

月度 LLM 成本 = 0.05 × 30 × 100,000 × 1.3 × 1.5 = **¥292,500**

### 第 3 层：上限熔断（hard cap）

设置自动熔断条件：
- 当月度成本超过预算 80% → 自动告警
- 当月度成本超过预算 100% → 自动降级（切换便宜模型）
- 当月度成本超过预算 120% → 自动暂停服务（避免无限烧）

**OpenAI / Azure 都原生支持 monthly budget hard cap**。配置好了再上线——不要等到 CFO 在群里 @ 你。

### CFO 能看懂的成本卡片模板

CFO 看不懂"5K input + 1K output"。CFO 看的是：

```
| 维度 | 数值 |
|-----|------|
| 月活用户（MAU）| 100,000 |
| 月均调用次数/用户 | 30 |
| 总调用次数 | 3,000,000 |
| 单次调用成本（含 retry）| ¥0.05 |
| 月度 LLM 总成本 | ¥150,000 |
| 月度 ARR（按订阅 ¥29.9）| ¥899,400 |
| LLM 成本占 ARR | 16.7% |
| 其他成本（服务器/带宽/客服）| ¥200,000 |
| 月度毛利 | ¥549,400 |
| 毛利率 | 61% |
| ⚠️ 头部 1% spike 风险 | +¥30,000-50,000 不确定成本 |
| ⚠️ 上限熔断阈值 | ¥300,000（触发自动降级）|
```

**给 CFO 的版本必须包括**：
- ✅ 用 RMB 不是 USD
- ✅ 用 MAU/月 不是 token/秒
- ✅ 显式标 retry 系数和头部 1% spike 风险
- ✅ 与传统 SaaS 对比毛利率
- ✅ 上限熔断阈值清楚标出来

## 上限熔断的业内最佳实践

### OpenAI 原生支持

- **Monthly Budget hard cap**：达到额度后所有 API 直接报错
- **RPM / TPM / RPD / TPD 多维 quota**：分钟级 / 日级 / 请求数 / token 数 都能限

### Azure OpenAI

- **quota-per-deployment**：每个 deployment 单独 quota
- 企业级常用做法：**给每个客户/租户单独 quota + 预算预警**

### 业内 best practice（Vellum / Traceloop）

「**连续 N 次错误熔断 + spike detection + per-user hard cap**」——大客户场景必备三件套。

具体配置：
- 连续 5 次失败 → 熔断该 endpoint 1 分钟
- 当前消费 vs 滚动 7 日基线 > 3 倍 → 触发 spike 告警
- 单用户单日成本 > ¥100 → 自动限流

## 下半篇 · engineering reality

### 3 种典型成本失控路径

#### 失控 1：上下文累积

20 步 agent loop，单步 1K token 输出：
- 直觉：累计 input = 20K
- 实际：累计 input = **210K**（二次方膨胀）

为什么？第 N 步的 input 包含前 N-1 步的累积。

**应对**：限制 agent 步数（≤ 10 步），或者主动 truncate 历史 context。

#### 失控 2：retry 爆炸

某团队配错 retry loop——**一夜烧 $72,000 OpenAI credit**。

具体机制：模型失败 → retry → retry 仍失败 → 再 retry → 无限循环。

**应对**：
- retry 上限严格设置（最多 3 次）
- 每次 retry 前增加 exponential backoff
- retry 失败后必须有 hard fail（不要继续）

#### 失控 3：对抗用户

恶意用户故意刷 token：
- 上传超长文档（每次烧 50K input）
- 触发递归调用
- 利用 prompt injection 让模型输出大量 token

**应对**：
- 单用户单日上限（如 ¥10）
- 单次请求 input/output 上限
- 异常行为检测（异常长输入 / 异常高频率）

### 真实事件：Cursor June 2025 billing 暴雷

2025 年 6 月，Cursor 未预警切换到 credit-based billing：
- 老用户从 "$20 心智模型" 突然变成 $200-$400 的 overage
- Hacker News 有人晒 "**$350 in a week**"——相当于月成本涨到 10-20 倍
- 7 月 4 日 CEO 公开道歉退款

**这是公开样本里最适合给 CFO 讲的"如果我们也这么干"故事**。

教训：
1. **billing 模式变更必须给老用户充分通知**
2. **新模式必须有透明的成本上限**
3. **意外溢出必须自动熔断而非自动扣款**

## 给 CFO 算账的话术翻译

CFO 不会问 token，会问 RMB。

### 翻译模板

❌ 不要说："我们用 GPT-5.1，input $1.25/M、output $10/M。"
✅ 改说："**我们的 LLM 成本是每月活跃用户 ¥3.2，毛利空间 ¥27**。"

❌ 不要说："context window 8K。"
✅ 改说："**单次对话上限 5000 字**，超过会被截断。"

❌ 不要说："prompt caching 命中率 74%。"
✅ 改说："**用了 cache 之后月度账单从 ¥45 万降到 ¥4.5 万**。"

❌ 不要说："retry 3 次成本翻 4 倍。"
✅ 改说："**1% 失败率会让月度成本多 ¥X**。"

**CFO 听不懂 token，但他听得懂 ¥3.2、5000 字、¥4.5 万**。

## CFO 必问的 5 个成本问题

PM 在 PRD 评审前，自己先回答这 5 题：

| # | CFO 会问 | PM 必须答得清的版本 |
|---|---------|------------------|
| 1 | 这个产品每月烧多少钱？ | ¥X / MAU，规模 X 时月成本 ¥X |
| 2 | 头部 1% 用户烧多少？ | ¥X，需 hard cap 控制 |
| 3 | 一旦突破预算怎么办？ | 自动熔断到 [备用模型 / 规则 / 暂停] |
| 4 | LLM 成本占 ARR 多少？ | X%（健康 < 30%，预警 > 30%，必死 > 50%）|
| 5 | 同行毛利率如何？ | ICONIQ 数据 41-52%，传统 SaaS 70-80% |

5 题答不全 → **不要去 CFO 那里**。

## 质量工程视角看成本预算 = AI 时代的资源预算

**8 年做软件质量工程的视角**——

任何生产级系统的设计都必须有 **resource budget**（资源预算）——CPU、内存、带宽、磁盘 IO。资源预算的核心动作是「**单位经济模型 + 监控阈值 + 自动熔断**」。

放到 AI 产品上，token 就是新一代"资源"——

- 单位经济模型 = 单用户月成本
- 监控阈值 = 月度 LLM 成本占 ARR < 30%
- 自动熔断 = monthly budget hard cap

传统软件 PM 会问"这个功能要多少服务器"——是基本动作。
AI 时代 PM 必须问"这个能力每月烧多少 token"——是新的基本动作。

**不会这道题 = 不是合格的 AI PM**。
