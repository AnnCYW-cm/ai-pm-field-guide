第二十五个"不以为然"——

**90% 的 PM 等公司转型才开始学 AI**。

不是 PM 没野心。是大多数 PM **以为"我公司还没上 AI 基础设施，我学了也用不上"**——结果 3 年过去，公司没转型，自己也没成长——**因为他们以为"PM 个人 Native 化"必须等"公司 Native 化"**。

读完这一篇你会知道——**这是第一季最大的错觉**。

**PM 个人 Native 化 vs 公司 Native 化，是两条独立赛道**。
- 公司 Native 化是慢赛道——PM 个人控制不了，可能 3-5 年才动一次
- 个人 Native 化是快赛道——**今天就能开始，3 年后就是行业稀缺资源**

如果你只看着公司这条慢赛道——你就被自己锁死了。

整个第一季 26 篇是给你的工具箱——**但工具箱怎么用，取决于你走哪条赛道**。

这一篇是收束。把 26 篇拧成一条主线——「**个人 Native 化 4 动作 + 公司 Native 化 4 阶段 + 两条赛道为什么必须分开走**」。

## 上半篇 · workflow

### 个人 Native 化的 4 个独立动作（不依赖公司）

这是终章最重要的话——**PM 不需要等公司任何资源**就能开始的 4 件事。

#### 动作 1：每天用 Claude Code 跑 1 个 prompt 实验

详见第 17 篇。

- 工具：Claude Code / Cursor / ChatGPT（任选）
- 不需要公司账号，自己付 $20/月
- 每天 30 分钟
- 验证下周要提的需求是否可行

3 个月后——你比 90% 公司里只会写 PRD 的 PM 强 10 倍。

#### 动作 2：给任何产品看 50 条 trace

详见第 20、21 篇。

- 工具：随便用一个支持 trace 查看的 AI 产品（ChatGPT / Notion AI / Lovable / Cursor）
- 不需要是你公司产品，看别人产品也算
- 每天看 50 条
- 训练自己的"AI 产品病理"嗅觉

6 个月后——你看 AI 产品的输出，能立刻判断这是哪类 bad case。

#### 动作 3：写一份带 eval case 的 PRD

详见第 12 篇。

- 工具：任何文档工具（Notion / Markdown / 飞书）
- 不需要公司同意，自己写一份存着
- 选你公司任何一个产品需求
- 给它配 50 条 eval case + 4 级 bad case 分级 + cost 卡片

3 个月后跳槽——这份 PRD 就是你的最强简历。

#### 动作 4：处理 100 个真实 bad case

详见第 21 篇。

- 渠道：自己用 ChatGPT 时记下每个 bad case + 写下"我会怎么修"
- 不需要公司允许，纯个人训练
- 累计到 100 个 → 你已经是行业里第 10% 懂 bad case 管理的 PM

**核心**：这 4 个动作**完全不需要你公司有 AI 基础设施**。你今晚就能开始。

### 4 层次自评（个人 Native 化的进度量表）

每个层次对应不同稀缺度。判断你现在在哪层——决定你下一步走多快。

| 层次 | 核心能力 | 个人 Native 动作完成度 | 市场稀缺度 |
|------|---------|-------------------|-----------|
| L1 AI 用户级 | 会用 ChatGPT / Claude | 动作 1 起步 | 低（人人都该会）|
| L2 AI 产品参与者 | 会写带 eval 的 PRD + 看 trace | 动作 1-3 都做 | 中（市场供不应求）|
| L3 AI 产品架构师 | 能搭 eval 体系 + 做降级决策 + 给老板讲业内框架 | 4 个动作全到位 + 主导过 100+ bad case | 高（市场极少）|
| L4 AI 时代的产品思想者 | 能定义"什么是好的 AI 产品" + 写出行业级 product note | L3 + 持续公开输出 2+ 年 | 极高（全行业 100 人以内）|

#### L1 → L2 的关键动作
1. 每天用 AI 工具完成至少 30% 工作
2. 写一份带 eval case 的 PRD（即使没人要求）
3. 用 Claude Code 做 3 个产品 prototype
4. 看 50 条 trace（任何产品都行）

时间：3-6 个月

#### L2 → L3 的关键动作
1. 主导一次 AI 产品的 pre-launch eval（5 大类）
2. 处理过至少 100 个真实 bad case
3. 做过至少 1 次降级决策
4. 给老板讲过 1 次 "为什么不上 AI"
5. 看过 1000+ 条 trace

时间：1-2 年

#### L3 → L4 的关键动作
1. 写出 1 篇影响行业的 product note
2. 主导过 1 个从 0 到 1 的 AI 产品（百万级 ARR）
3. 给业内大公司做过 AI 产品咨询
4. 在公开场合（公众号 / 演讲 / 书）持续输出 2 年以上

时间：3-5 年

## 下半篇 · engineering reality

### 公司 Native 化的 4 个阶段（PM 帮老板看清）

PM 个人在快赛道狂奔——但公司在慢赛道，PM 还要帮老板推动。

详见第 25 篇 AI Native 团队组建。这里只给 PM 的"老板对齐"压缩版：

#### 第 1 阶段（1-3 月）：试点
- 选 1 个团队（如客服）做 AI 武装
- 不裁人，让团队用 AI 工具
- 观察产能变化 + 客户满意度变化

#### 第 2 阶段（3-6 月）：扩展
- 把试点经验推到 2-3 个团队
- 新招 1 个 Eval Engineer
- 让 PM 和工程师都开始用 Claude Code / Cursor

#### 第 3 阶段（6-12 月）：组织重塑
- 招 AI Ops 负责整体监控
- 把 QA 团队转型为 Evaluator
- 业务扩张时优先用 AI 提产能

#### 第 4 阶段（12-18 月）：成熟
- 整个公司形成"人 × AI = 5-10 倍产能"的运行模式
- 人均 ARR 显著高于竞品

**关键**：整个路线图**没有"砍人"步骤**——参见第 25 篇 Klarna 反例。

### 两条赛道为什么必须分开走

很多 PM 把"公司转型"和"个人成长"绑在一起——这是 AI 时代 PM 最大的认知错位。

| 维度 | 公司 Native 化（慢赛道）| 个人 Native 化（快赛道）|
|------|-------------------|-------------------|
| 时间 | 1-2 年起步，3-5 年成熟 | 今天就能开始，3 年到 L3 |
| 控制权 | 90% 在老板手里 | 100% 在你手里 |
| 依赖 | 需要预算、人事、合规、文化 | 只需要 $20/月 Claude 订阅 |
| 风险 | 慢、被动、可能停摆 | 完全自主 |
| 回报 | 公司转型成功 → 你也受益 | 个人 L3 后 → 跳槽议价 5-10 倍 |
| 不绑定 | 公司不转型 → 你的能力不受影响 | 个人转型 → 比公司快 3-5 年 |

**最关键的判断**——

**如果你公司在快赛道**（已经在做 AI 产品）——两条赛道并行，公司提供实战场景，你的个人能力直接进入 L2-L3。

**如果你公司在慢赛道**（还没启动 AI 转型）——你必须把个人 Native 化优先级**抬到最高**——3 年后跳槽，你比同龄人有 5-10 倍议价空间。

**最危险的状态**——公司不转，你也不转，等着风口过来——3 年后你既没有公司平台，也没有个人能力，**这是 AI 时代 PM 最大的失败模式**。

### 「不依赖公司」成长的 3 类样本

业内反复观察到的模式——3 类 PM 用这套打法在 1-2 年内拿到议价空间显著高于同龄人的 offer：

#### 样本 1：传统 SaaS 转 AI 应用层
- **起点**：传统 SaaS 公司 5 年 PM 经验，公司未启动 AI 转型
- **个人动作**：每天 1 小时用 Claude Code 跑 prompt 实验 + 写带 eval 的 PRD 作品集 + 在公开渠道（小红书 / 即刻 / X）持续输出 12 个月以上
- **跳槽节点**：议价倍数普遍在 1.5-2.5 倍区间（业内招聘端口径，具体因地域 + 阶段差异）

#### 样本 2：B 端工具厂转生产级 AI 创业
- **起点**：B 端工具厂中级 PM，已用过 Claude Code / Cursor 做内部 prototype
- **个人动作**：主导过至少 1 次完整 pre-launch eval（5 大类，详见第 19 篇） + 处理过 100+ 真实 bad case + 给老板讲过降级决策（详见第 22 篇）
- **跳槽节点**：直接跳进 AI 创业公司 L3 岗位（参见前面 L3 定义——AI 产品架构师）

#### 样本 3：QA / 测试架构师转型 AI PM
- **起点**：QA / 测试架构师背景，无传统 PM 经验
- **个人动作**：用 FMEA 视角写 PRD（详见第 11 篇 PRD 4 加 2 砍）+ 主导 eval 体系搭建 + 把质量工程 mindset 应用到 AI 概率系统
- **跳槽节点**：直接进 AI PM L2-L3，**质量工程背景成为差异化锚点**（这条路径就是本书作者自己走过的——8 年 QA → AI PM）

### 3 类样本的共性

观察 3 个样本——共性有 3 条：

1. **不等公司提供 AI 项目** —— 个人订阅 + $20/月就能开始
2. **用「1 小时/天 × 12 个月」积累可验证作品集** —— 不是听课，是产出
3. **公开输出 thought leadership** —— 让招聘方主动找你，不是你找招聘方

### 招聘方的明确信号

业内招聘端反复出现的信号——

> **拿出 AI PM 实战作品集的候选人，议价空间显著高于那些只能说「我公司没让我做 AI」的候选人。**

为什么？因为 AI 公司知道——
- 「公司没让你做」≠「你没能力做」
- 作品集 = 可验证的能力证明
- 公开 thought = 可验证的判断力证明

**这是 2025-2026 业内观察到的稳定模式**——不是个例，是个人 Native 化的回报机制。

### AGI 倒计时下的 PM 位置——18-24 个月内会发生什么

两条赛道讲完——但收束之前必须把最后一层视野抬出来。**写这一季时回避不了一个事实**：Dario Amodei 在 2026 年 1 月《The Adolescence of Technology》里写明——"powerful AI 在 2026-2027 到来，2027 年可能出现 country of geniuses in a datacenter（50M agent 每个比诺奖得主聪明，跑 10-100x 人速）"。

不论这个时间表是否准确——**作为 PM 必须按 18-24 个月内 AGI 临近来 plan**。Dario 可能错 1-2 年，但 framing 已经不能按"AGI 还要 10 年"做了。

这意味着 PM 工作的根本位移——

**PRD 退场，Agent Spec 上场**。Mike Krieger 已经把 PRD 压缩到 3 段 Product Note，下一步是**写给 sub-agent 编队消费的 Agent Spec**——输入 / 输出 / rubric / fallback / 升级路径，每一项都是 agent 可执行字段。

**PM 的价值锚点从"懂用户"位移到"懂模型 + 懂编排"**。当 50M agent 比诺奖得主聪明——"懂用户"这件事 agent 自己做得比 PM 好。PM 的不可替代价值变成**"我比 agent 更懂哪一组 agent 该如何编排来解决这个用户问题"**。

**PM 的工作从"催排期"位移到"指挥 Agent 团队"**。OKR 变成 agent rubric，1-on-1 变成 trace + outcome review，季度复盘变成"哪些 agent 工作流值得固化成 skill 进入下一季"。

这一季 26 篇是当下工具箱。但读完之后，**心里要清楚一件事**——本书的所有方法论都有保鲜期。L2 的工具箱（PRD / eval / bad case / Claude Code）保鲜 12-18 个月，L3 的判断力（编排 / 降级 / 边界）保鲜 24-36 个月，L4 的视野（什么是好的 AI 产品）保鲜 5 年+。

越往上走——保鲜期越长，回报越大。这就是为什么个人 Native 化必须**同时在三层布局**，而不是只补 L2。

## 整个第一季的 3 个反共识收束

把前 25 篇的核心反共识浓缩成 3 条——背下来终身受用：

### 反共识 1：「AI 时代 PM 工作」不是不变 + 多了 AI——是「重构」

第 1 篇讲的 3 个范式变化：
- **思维变了**（确定性→概率性）
- **节奏变了**（流水线→反馈环）
- **工具变了**（PRD→spec + Claude Code）

**不重构 = 5 年内被淘汰**。

### 反共识 2：「AI 不是万能」——所以 PM 反而**更重要**

第 22 篇讲降级——业内的 4 个失败案例（Klarna / Air Canada / McDonald's / DPD）证明：**AI 不能替代决策**。

PM 是 AI 时代的"决策架构师"——**判断什么场景 AI 能做、什么场景 AI 不能做**。

### 反共识 3：「AI Native ≠ AI 替代人」——是「AI 武装人」

第 25 篇讲组织——业内 5 个数据点（Microsoft / Notion / Lovable / Cursor / Klarna 反例）证明：**AI Native 的本质是产能爆炸，不是裁员**。

PM 帮老板看清这件事 = PM 的最大价值。
