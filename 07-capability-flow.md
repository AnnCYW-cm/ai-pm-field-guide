# 07 · AI 产品方案设计的范式切换——从功能流到能力流

> 第一季《AI 时代 PM 工作流重构》· 第 7 篇 · 第二部分 方案
> 预估字数：5000

---

第六个"不以为然"——

**90% 的 PM 还在用「功能流图」评审 AI 产品**。

不是 PM 不想换图。是大多数 PM **以为「图变不变没那么重要，反正最后还是评审通过开干」**——结果工程师看完图心里在默默重画一张"能力流图"再去开发，**沟通成本翻 3 倍**，上线后才发现"图没说的部分全部出问题"。

读完这一篇你会知道——**传统 PM 画功能图（A 按钮→B 页面→C 接口），AI PM 画能力图（能力 1+能力 2+能力 3=用户体验）。范式切换从画图开始**。

## 上半篇 · workflow

### 功能流图 vs 能力流图：图示对比

**功能流图**长这样：

```
[用户登录] → [输入查询] → [点击搜索按钮]
            → [返回结果列表] → [点击详情] → [展示详情页]
```

每个箭头是一个确定的跳转。每个方框是一个确定的功能。

**能力流图**长这样：

```
[用户输入]
    ↓
[意图识别能力]（准确率 92%，p99 延迟 800ms，单次成本 ¥0.003）
    ↓
[根据意图分流]
    ├→ [信息检索能力]（召回率 85%，依赖 GPT-5）
    │       ↓
    │   [答案生成能力]（依赖 retrieval 结果，幻觉率 < 3%）
    │       ↓
    │   [置信度判断]
    │       ├→ 高置信 → 直接返回
    │       └→ 低置信 → fallback 到人工
    │
    └→ [行动执行能力]（工具调用 ≤ 3 步）
            ↓
        [失败检测]
            ├→ 成功 → 返回结果
            └→ 失败 → 重试 1 次 → 失败 → 降级到规则
```

**根本差异**：
- 功能流图标的是「**跳转关系**」，能力流图标的是「**能力 + 可靠性 + 成本 + 降级路径**」
- 功能流图的每个方框是确定的，能力流图的每个方框是**概率分布**
- 功能流图能用 PowerPoint 画，能力流图必须用支持注释的工具（Notion / Excalidraw / Mermaid）

### 能力流图的画法 4 步

#### 第 1 步：识别核心能力

把需求拆成最小能力单元。每个能力满足：
- 单一职责（只做一件事）
- IO 明确（输入是什么、输出是什么）
- 可独立评估（有自己的 evals）

例：「自动整理会议纪要」拆成：
- 语音转文字能力
- 说话人分离能力
- 关键点提取能力
- 摘要生成能力
- 待办事项提取能力

**5 个独立能力，不是 1 个"自动整理会议纪要"功能**。

#### 第 2 步：标注能力可靠性

每个能力旁边标 3 个数字：
- **准确率**（基于 100+ case 的 eval 结果）
- **p99 延迟**（最慢的 1% 请求要多久）
- **单次成本**（¥/次）

如果你这 3 个数字标不上来——这个能力**还没准备好画图**。回去先做能力探针（第 3 篇）。

#### 第 3 步：标注能力依赖

显式画出哪些能力依赖哪些。

例：摘要生成能力依赖语音转文字能力的输出 → 如果转文字错了，摘要必然错。

**依赖关系决定了故障传播路径**——上游能力出错，下游能力会跟着出错。

#### 第 4 步：标注能力降级路径

每个能力旁边标"如果失败怎么办"：
- 重试 N 次
- 降级到更便宜的模型
- 降级到规则
- fallback 到人工
- 直接报错让用户重试

每个能力都必须有降级路径。**没有降级路径的能力 = 上线后必崩**。

## 用能力流图开方案评审会

传统功能图评审会的话术：

> PM：「这里点 A 按钮跳到 B 页面。」
> 工程师：「OK，多久要做完？」

能力流图评审会的话术：

> PM：「这里调用意图识别能力。准确率 92%，错的 8% 走兜底流程。p99 延迟 800ms，超时降级到规则。单次成本 ¥0.003，月活 10 万的话每月成本 ¥X。」
> 工程师：「OK，准确率 92% 这个数怎么测的？eval 集多大？」
> PM：「200 个 case，3 个标注员独立标，IRR ≥ 0.85。」
> 工程师：「行，开做。」

**两种评审的根本差异**：
- 功能图评审讨论的是"**什么时候做完**"
- 能力图评审讨论的是"**做出来什么样算合格**"

**第二种讨论让 AI 项目能真正闭环**。

## 下半篇 · engineering reality

### 现实 1：为什么功能流图对 AI 产品有误导性

功能流图的底层假设：**确定性系统**。

每个方框、每个箭头都默认 100% 工作。如果失败，是 bug，应该修复。

AI 产品的底层是**概率性系统**。每个能力都有失败概率，**失败不是 bug 是常态**。

功能流图把概率性系统画成确定性系统——**它隐藏了 AI 产品最重要的设计变量**：可靠性分布、成本分布、延迟分布。

工程师看着功能图心里在算"这个箭头要 retry 几次、这个方框可靠性多少、超时降级到哪"——**他在脑子里默默重画一张能力图**。

如果你直接给他能力图——你和他从一开始就在看同一张图。**沟通成本降到 1/3**。

### 现实 2：能力可靠性、成本、延迟必须独立测算

传统 PM 估算：
> 「这个功能做完大概需要 2 周。」

一个数字。

AI PM 估算：
> 「意图识别能力准确率 92% / p99 800ms / 成本 ¥0.003。
>  信息检索能力召回率 85% / p99 1200ms / 成本 ¥0.005。
>  答案生成能力幻觉率 < 3% / p99 2000ms / 成本 ¥0.012。」

**3×3 = 9 个数字**。

为什么？因为**能力可靠性 / 成本 / 延迟没法像功能一样"打包估算"**。每个能力的瓶颈完全不同——意图识别可能是延迟瓶颈，答案生成可能是成本瓶颈，信息检索可能是召回率瓶颈。

**打包估算 = 隐藏了真正的瓶颈**。上线后才发现"原来是答案生成成本爆了"——已经晚了。

### 现实 3：能力之间的耦合是工程现实

A 能力的优化可能让 B 能力下降。**这不是 bug，是工程现实**。

#### 学术背书
EMNLP 2024 一篇 RLHF 对齐税论文实测：OpenLLaMA-3B 偏好对齐奖励从 0.16 提到 0.35 时——SQuAD F1 **掉 16 分**、DROP F1 **掉 17 分**、WMT BLEU **掉 5.7**。

「安全能力上升 = 翻译/阅读理解能力下降」**是可量化的**。

#### 行业实证
Cursor 团队公开承认：**Claude 3.7 在 Claude Code 上 post-train 后，编码大幅提升但通用能力下降**。

#### 设计含义
能力流图必须能体现这种耦合。建议画法——
- 用**虚线**连接"会互相影响"的能力
- 在虚线上标注"如果优化 A，B 可能下降 X%"
- 评审会讨论"我们要不要为了 A 牺牲 B"

**如果你的能力流图没画虚线——你的方案设计忽略了 AI 产品最难的工程现实**。

## 质量工程视角看能力流图

**8 年做软件质量工程，反复见过一个模式**——

任何复杂系统的设计图都需要标注 3 件事：**failure mode（失败模式）、blast radius（影响范围）、mitigation（缓解策略）**。

能力流图本质上就是 AI 产品的 **failure mode 可视化**——
- 每个能力的"准确率 92%" = 标注失败模式
- 每个能力的"依赖关系" = 标注影响范围（A 崩了 B 也崩）
- 每个能力的"降级路径" = 标注缓解策略

传统软件设计图也有这 3 件事，只是因为确定性系统的失败模式较少，大家习惯省略。

**AI 产品把失败模式从"少数边界 case"放大到"概率分布的常态"——所以能力流图必须显式画出这 3 件事**。

不画 = 设计图回到 20 年前确定性系统的水平，应付不了 AI 时代的复杂度。

## 业内「能力流图」现状：未验证有统一术语

诚实告诉你——**业内目前没有统一的"能力流图"术语和模板**。

Anthropic 有 Claude Design 哲学（progressive disclosure），Cursor 有"engine vs car"论，Anthropic 的 AI-First Dev 工作流有 plan/act/review 三段——**这些都是相关的方法论，但没有一家叫它"能力流图"**。
