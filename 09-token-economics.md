## 上半篇 · workflow

### Token 账本 3 层结构

#### 第 1 层：单次调用成本

每次用户与 AI 交互的成本：

```
单次调用成本 =
  (input_tokens × 输入单价) +
  (output_tokens × 输出单价) +
  (cache_read_tokens × cache 单价 - 节省) +
  (其他附加成本：embedding / 工具调用 / 第三方 API)
```

⚠️ 关键提醒：**output 比 input 贵 3-10 倍**。**这是 CFO 最常被打脸的点**。

典型 chatbot 实算：
- input：1M token × $0.15 = $0.15
- output：2M token × $0.60 = $1.20
- 合计：**$1.35/M tokens**——比官网"$0.15"贵 9 倍

#### 第 2 层：用户活跃成本

```
单用户月成本 =
  单次调用成本 × 月均调用次数 × (1 + retry 系数)
```

retry 系数是大坑——失败 retry 1 次 = 成本翻倍。

例：客服 chatbot 单次调用 ¥0.05，用户月均 30 次对话，retry 系数 1.3：
- 单用户月成本 = 0.05 × 30 × 1.3 = **¥1.95/月**

#### 第 3 层：月度预算上限

```
月度预算 = 单用户月成本 × DAU × 头部 1% spike 系数
```

头部 1% spike 系数是另一个大坑——**1% 头部用户烧掉 99% 的 token**。

例：DAU 10 万，单用户月成本 ¥1.95，头部 1% spike 系数 1.5：
- 月度预算 = 1.95 × 100,000 × 1.5 = **¥292,500**

如果你的产品收 ¥29.9/月，毛利空间是：
- ¥29.9 - ¥1.95 = ¥27.95
- 毛利率 93%——**看起来很美**

但如果有 1% 头部用户烧 ¥50/月（聊天狂魔）：
- 这 1% 的毛利 = ¥29.9 - ¥50 = **-¥20.1（负）**
- 99% 普通用户的盈利还要补这 1% 的亏损

**这就是为什么你产品看起来盈利，CFO 总在问"为什么钱还是不够"**。

### 反推工作流：从 ARPU 倒推预算

正向算（先选模型再算成本）：
- 选 Opus → 算成本 → 算价格 → 算毛利

**错的**。

反向算：
- 用户能付多少钱（ARPU）
- 你想要多少毛利率（如 50%）
- 倒推单次调用预算
- 倒推上下文长度上限
- 倒推选哪个模型

例：
- ARPU ¥29.9/月，毛利目标 50%
- 单用户月成本上限 ¥15
- 月均 30 次对话
- 单次调用预算 ¥0.5
- 按 Sonnet $3 输入 / $15 输出 → 单次最多 5K input + 1K output
- 这意味着 system prompt 必须用 cache（否则一次就超预算）

**反推让你在选模型之前就知道商业模式是否成立**——而不是上线后才发现"卖一单亏一单"。

### CFO 能看懂的 token 账本模板

CFO 看不懂"5K input + 1K output"。CFO 看的是：

```
| 维度 | 数值 |
|-----|------|
| 月活用户（MAU）| 100,000 |
| 月均调用次数/用户 | 30 |
| 总调用次数 | 3,000,000 |
| 单次调用成本（含 retry）| ¥0.05 |
| 月度 LLM 总成本 | ¥150,000 |
| 月度 ARR（按订阅）| ¥899,400 |
| LLM 成本占 ARR | 16.7% |
| 其他成本（服务器/带宽/客服）| ¥200,000 |
| 月度毛利 | ¥549,400 |
| 毛利率 | 61% |
| ⚠️ 头部 1% spike 风险 | +¥30,000-50,000 不确定成本 |
```

**给 CFO 的版本必须包括**：
- 用 RMB 不是 USD
- 用 MAU/月不是 token/秒
- 显式标 retry 系数和头部 1% spike 风险
- 与传统 SaaS 对比毛利率（传统 70-80%，AI 产品当前 ICONIQ 数据 41-52%）

### 商业模式预警信号

> **当 LLM 成本占 ARPU 超过 30% 时，模式就有问题**。

为什么 30%？因为传统 SaaS 还有服务器、带宽、客服、销售、运营、税——LLM 之外还要至少 30-40% 成本。

如果 LLM 占 30%，留给其他成本和毛利的只剩 60-70%。**毛利率压不到健康区间**。

> **当 LLM 成本占 ARPU 超过 50% 时，模式必死**。

例：
- ARPU ¥29.9，LLM 成本 ¥18 → LLM 占 60% → **必死**
- ARPU ¥199，LLM 成本 ¥30 → LLM 占 15% → 健康

如果你的项目当前是第一种——**赶紧改商业模式或砍能力，不要等到上线后让 CFO 痛骂**。

## 下半篇 · engineering reality

### 5 个常见 token 漏算

#### 漏算 1：上下文累积的二次方膨胀

20 步 agent loop，单步 output 1K token：
- 直觉：累计 input = 20K
- 实际：累计 input = **210K**（二次方膨胀）

为什么？因为第 N 步的 input 包含前 N-1 步的累积。
- 第 1 步 input = 0
- 第 2 步 input = 1K
- 第 3 步 input = 2K
- ...
- 第 20 步 input = 19K
- 总 input = 0 + 1 + 2 + ... + 19 = **190K**（加上原始 1K = 191K，约 210K）

如果你的 agent 有 20+ 步——**成本是单步的 20 倍以上**。

#### 漏算 2：retry 把错误信息塞进 context

retry 不是"重新算"，是"带着错误信息再算"。

每次 retry 把上次失败的输出 + 错误信息 + 原始 input 一起塞进 context——**越 retry 越贵**。

3 次 retry 的成本可能是首次的 5-10 倍。

#### 漏算 3：fallback 成本

PM 设计"主模型失败 → fallback 到备用模型"。

这意味着 1 次用户请求触发**两次模型调用**——主模型 + fallback 模型。

如果你的主模型失败率 10%，fallback 成本 = 10% × 备用模型单次成本。

如果备用模型更贵（如主模型用 Haiku，fallback 到 Opus）——**fallback 成本可能比主模型还高**。

#### 漏算 4：缓存 miss

Prompt caching 听起来美好。但**缓存有 5-10 分钟过期时间**。

如果你的用户行为是"间隔 30 分钟问一次"——**每次都是 cache miss**，缓存等于没用。

需要按业务场景算"实际缓存命中率"。ProjectDiscovery 案例：把动态内容挪出 cacheable prefix 后，命中率从 7% 提到 74%——**6 倍差距**。

#### 漏算 5：超长输入截断后二次调用

超长 input 模型会截断。一些 PM / 工程师设计"超长就拆成 2 次调用"。

意思是：1 次用户请求 → **2 次模型调用 + 2 次 input** → 成本翻倍。

如果你的产品有"上传长文档"功能——**这是隐藏的最大成本黑洞**。

## 给 CFO 算账的话术

CFO 不会问 token，会问 RMB。

**翻译模板**：

❌ 不要说："我们用 GPT-5.1，input $1.25/M、output $10/M。"
✅ 改说："我们的 LLM 成本是**每月活跃用户 ¥3.2**，毛利空间 ¥X。"

❌ 不要说："context window 8K。"
✅ 改说："**单次对话上限 5000 字**，超过会被截断。"

❌ 不要说："prompt caching 命中率 74%。"
✅ 改说："**用了 cache 之后月度账单从 ¥45 万降到 ¥4.5 万**。"

**CFO 听不懂 token，但他听得懂 ¥3.2、5000 字、¥4.5 万**。

## 质量工程视角看 token 账本

**8 年做软件质量工程的视角**——

任何生产级系统的设计都必须有 **resource budget**（资源预算）——CPU、内存、带宽、磁盘 IO。资源预算的核心动作是「**单位经济模型 + 监控阈值 + 自动熔断**」。

放到 AI 产品上，token 就是新一代"资源"——

- 单位经济模型 = 单用户月成本
- 监控阈值 = 月度 LLM 成本占 ARR < 30%
- 自动熔断 = monthly budget hard cap

**5 个漏算的本质，都是因为 PM 把 token 当"工程师的事"而不是"自己的资源预算"**。

传统软件 PM 会问"这个功能要多少服务器"——是基本动作。
AI 时代 PM 必须问"这个能力每月烧多少 token"——是新的基本动作。

不会这道题 = **不是合格的 AI PM**。
