# 05 · Agent 能力边界——哪些需求"看着能做"其实做不出来

> 第一季《AI 时代 PM 工作流重构》· 第 5 篇 · 第一部分 需求
> 预估字数：5000

---

第四个"不以为然"——

**90% 的 PM 觉得「Agent 能干什么是工程师的事，PM 不用懂」**。

不是 PM 不想懂。是大多数 PM **以为「先立项再让工程师评估能不能做」就够了**——结果立项后 2 个月，工程师告诉你"做不出来"，团队季度预算烧光。

读完这一篇你会知道——Agent 有 **4 类硬边界**，越过任何一类，需求就从"难"变成"做不出来"。**PM 必须在需求阶段就识别这 4 类边界**，否则上线第一周就要回滚。

---

## 上半篇 · workflow：4 类 Agent 能力边界

### 边界 1：推理深度

**判断标准**：单步推理超过 3-4 跳逻辑，准确率会断崖式下降。

什么叫"跳"？例如：

- "分析这家公司的财报" = 1 跳
- "对比这家公司和竞品的财报" = 2 跳
- "根据财报对比，判断哪家更适合投资" = 3 跳
- "根据投资判断，给出具体仓位建议和退出策略" = 4 跳

每多一跳，需要的"工作记忆"就翻一倍。模型在第 3 跳之后开始**幻觉式融合**——它不告诉你它没想清楚，它编一个看起来合理的答案。

如果你的需求"一句话里包含 4 个以上动词"——**基本就是越过了推理深度边界**。

**应对**：把需求拆成多步。每一步只让模型推理 1-2 跳。**把"链"显式化**。

### 边界 2：工具调用复杂度

**判断标准**：工具调用超过 5 步，基本不要做。

这条不是拍脑袋。**Anthropic 自己的工程数据**：

- **58 个工具就吃掉 ~55k token**
- 当工具池增大，模型选对工具的概率显著下降
- **官方建议**：30+ 工具就要做 Tool Search，否则崩

更精确的推算：单步选对工具的准确率约 80%，**5 步连续成功概率仅 32.8%**（80%^5）。

如果你的需求"需要 5 个以上工具协同"——准确率掉到 1/3 以下。**这不是产品，是赌博**。

**应对**：要么减少工具数（合并 / 删减），要么做 Tool Search（让模型先选小池再决策），要么把多步流程拆成多个独立 Agent。

### 边界 3：长链条规划

**判断标准**：任务时长超过 30 分钟（人类基线），Agent 失误概率指数级上升。

METR（独立 AI 评估机构）的 "horizon length" 系列研究：前沿模型能稳定完成的任务时长**每 7 个月翻倍**。听起来很乐观对吧？

但反过来看——任何长链条任务，单步错误率累乘。到第 N 步成功率 = p^N。

p = 95%（已经很高了）：
- N=10：成功率 60%
- N=20：成功率 36%
- N=50：成功率 8%

这就是为什么 OSWorld 这个 computer-use benchmark 给出的数据这么尴尬：

- **OpenAI Operator (CUA) 38.1%**
- **Anthropic Computer Use 22.0%**
- **人类基线 72.4%**

不是模型不行——**是长链条任务对单步精度的要求高到目前模型达不到**。

**应对**：不要做"全自动化"需求。做"AI 完成 70% + 关键节点人工确认"的混合流。

### 边界 4：跨系统协调

**判断标准**：需要协调 3 个以上独立系统的数据 + 操作，几乎必崩。

为什么？因为：
- 每个系统的数据 schema 不一样（需要适配层）
- 每个系统的延迟不一样（需要等待 + retry）
- 每个系统的失败模式不一样（需要分别处理）
- 系统之间的数据状态不一定一致（race condition）

PM 看着是"AI 自动跨系统执行任务"，工程师看着是 N 个不可靠组件的串联——**任何一个崩，整个流程崩**。

**应对**：把跨系统需求拆解成"AI 推荐 + 人工执行"。AI 给方案，人点执行键，每一步都有 audit log。

## 边界识别工作流

把 4 类边界串起来，需求阶段你应该这样判断：

```
Step 1：把需求画成 Agent 思考链草图
  → 列出从输入到输出的所有步骤
  → 每一步标注：推理跳数 / 工具数 / 是否跨系统

Step 2：4 类边界逐个检查
  → 边界 1：单步推理 ≤ 3 跳？
  → 边界 2：总工具数 < 5？
  → 边界 3：人类完成总时长 < 30 分钟？
  → 边界 4：跨系统数 < 3？

Step 3：任何一项越界 → 设计降级
  → 推理深度越界 → 拆多步
  → 工具复杂度越界 → 用 Tool Search 或拆 Agent
  → 长链条越界 → 加人工确认节点
  → 跨系统越界 → 改成"AI 推荐 + 人工执行"

Step 4：降级后重新评估
  → 还能不能闭环？
  → 商业上还能不能算账？
  → 不能闭环 → 砍掉这个需求
```

**15 分钟过一遍，比上线后回滚省 3 个月**。

## 替代方案：「Agent + 规则」组合降级

当一个需求越过边界，你有 3 种替代方案。

### 方案 1：拆成多个独立 Agent

把"全自动办公 Agent"拆成"自动整理邮件 Agent" + "自动安排日程 Agent" + "自动写周报 Agent"——3 个独立 Agent 各自完成单一任务，**准确率回到可用区间**。

### 方案 2：AI 决策 + 规则执行

用 AI 做"判断"，用规则做"执行"。

例：合同审核需求——AI 标记 20 个常见风险点（判断），规则按风险等级路由到不同审批层级（执行）。**AI 错了不会直接出事，规则兜住**。

### 方案 3：人工 in the loop

Agent 跑到关键决策节点停下来，等人类确认，再继续。

例：自动客服 Agent 处理退款——AI 判断"该退款"，但实际退款动作需要人工点击确认。**AI 错判不会真的退钱**。

3 种方案的共同点：**把"AI 自主"改成"AI 辅助"**。这不是退步，是负责任。

---

### 4 类边界在未来 24 个月会怎么变

上面 4 类边界是 **2026-Q2 模型快照**——不是常量。给读者一份"边界位移速查"：

- **推理深度 3 跳**：Anthropic 2026 推出的 Outcomes（独立 grader 判每个中间结果是否达标，不达标自动 revise）正在把"单次 forward pass 的工作记忆上限"打破——**3 跳边界在 12 个月内可能位移到 5-6 跳**。
- **工具调用 5 步**：Tool Search 已经从论文进入产品。58 个工具吃 55k token 的痛点在 Skills 形态下被绕过——**Skills 是按需 load 的小包，5 步边界 6 个月内会被"按需 skill 调用"替代**。
- **长链条 30 分钟**：METR Time Horizon 1.1（2026-01）显示 doubling time 已经从 7 个月缩到 4.3 个月。Claude Opus 4.6 单任务 50% horizon 已到 14.5 小时，2026-05 Claude Mythos 到 16 小时——**"30 分钟"这条边界在本书出版的 12 个月内就会过时**。
- **跨系统 3 个**：Sub-agent + Memory + 共享 filesystem 正在把"跨系统协调"变成"sub-agent 之间的内部协调"——本质不再是分布式一致性问题。

读者拿走的不是"4 个数字"，是**"边界每 6 个月重测一次"的纪律**。不重测的 PM——会用 2026-Q2 的数据训练自己到 2028 还在用。

## 下半篇 · engineering reality：4 类边界为什么存在

### 为什么有推理深度边界

LLM 的"思考"是一次性 forward pass——它不像人类可以反复推敲。链式推理（chain-of-thought）是模拟"思考过程"，但本质还是一次 token 生成。

每多一跳推理，需要的中间 token 就线性增加。当 token 数超过模型的"工作记忆"（通常 4K-8K 有效推理窗口），**模型开始遗忘前面的步骤**。

它不会告诉你"我忘了"，**它会编一个看起来合理的衔接**。这就是"幻觉式融合"。

### 为什么有工具调用复杂度边界

Anthropic 在 2025-2026 年的工程博客里讲得很细：**58 个工具就吃掉 ~55k token**。意味着模型每次调用都要先"看完工具列表"再决策。

工具越多，干扰越大。Anthropic 的实测：Claude Opus 4.5 上 Tool Search 把准确率从 79.5% → 88.1%——意味着不做 Tool Search 时，**模型在大工具池里选错的概率接近 20%**。

5 步连续 80% 选对的概率 = 80%^5 = 32.8%。这就是为什么"5 步以上不要做"。

### 为什么有长链条规划边界

Partnership on AI 2025 年发表的 *Prioritizing Real-Time Failure Detection in AI Agents* 把"实时失败检测"列为长 Agent 部署的核心命题——**业内默认 Agent 会失败**。

研究的本质问题是：怎么在 Agent 跑到第 30 步时知道它已经走偏了？答案是——**你不知道**。Agent 不会主动告诉你它在第 12 步已经迷路了，它会继续编故事到第 30 步给你一个看似合理但完全错误的结果。

这是为什么"长 Agent 的工程美学不是让它跑得更远，是让它在第 3 步就乖乖停下来求救"。

### 为什么有跨系统协调边界

跨系统协调的本质是分布式一致性问题——这是计算机科学几十年都没解决的难题。让 LLM 来"解决"，等于让一个连数学题都常错的高中生去当 CTO。

每多一个系统，状态不一致的概率就上升一个维度。3 个系统的协调 = 8 种可能状态组合 = 8 种 corner case。**LLM 没法穷举处理**。

## 质量工程视角看 Agent 边界

**8 年做质量工程，反复见过一个模式**——

任何"看似全自动"的系统，**都有一个或几个隐性失败点**。工程师的工作就是把这些失败点找出来、量化、监控。

Agent 系统的 4 类边界，本质是 4 类隐性失败点：
- 推理深度边界 = 工作记忆失败点
- 工具调用复杂度边界 = 决策失败点
- 长链条规划边界 = 累积错误失败点
- 跨系统协调边界 = 状态一致性失败点

PM 在需求阶段画"Agent 思考链草图"——本质是做一次 **FMEA（Failure Mode and Effects Analysis）**。这是传统质量工程的基本动作，**LLM 圈把它丢了——只是因为 Agent 看起来"什么都能做"，让人误以为可以跳过失败模式分析**。

如果你的 Agent 类需求 PRD 里没有"4 类边界自检"+"降级路径"两章——**你做的不是 AI 产品，是赌博**。

---

## 模型升级会不会消除这些边界

很多 PM 安慰自己——「现在做不到，等模型升级就能做到了」。

**部分对，部分错**。

**对的部分**：模型每代升级，单步精度在提升。METR 数据显示前沿模型完成任务时长每 7 个月翻倍。

**错的部分**：边界本质是"指数 vs 线性"的赛跑。**模型升级是线性提升单步精度，但任务复杂度是指数增长的累乘**。

具体到数字——
- 单步精度从 80% 提到 90% 只能让 5 步成功率从 32.8% 提到 59%
- 单步精度从 90% 提到 95% 只能让 5 步成功率从 59% 提到 77%

**真正解决问题的不是模型升级，是产品设计降级**。

PM 需要的不是"等模型变强"，是「**现在就识别边界并设计降级**」。

## Manus AI 的真假壁垒

最后一个案例。

Manus（蝴蝶效应公司出品）当初宣称 GAIA benchmark SOTA、估值传闻在数十亿美元量级。

业内多家技术媒体 2025 年初实测后给出的共识反馈——
- benchmark 漂亮
- 但实际有 reliability、上下文窗口、付费墙、CAPTCHA 全卡死的问题
- 频繁崩溃

**benchmark 漂亮，真实任务卡 CAPTCHA**。

这就是 4 类边界的综合体现——长链条规划崩、跨系统协调崩、工具复杂度崩。

**核心判断**：Manus 的真技术壁垒在 4 类边界面前其实不存在——benchmark SOTA ≠ 生产可用，这是所有 Agent 类产品的通病。
