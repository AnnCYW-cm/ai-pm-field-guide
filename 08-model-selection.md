### 选型工作流：「够用就好」原则

PM 圈最常见的选型错误是"先选最强的"。

正确做法是反向——**从最便宜的模型开始往上加**。

```
Step 1：用最便宜的模型（Haiku / GPT-5 / Gemini 3 Flash / DeepSeek V3.2）
  → 跑自己的 eval 集（50-100 case）
  → 看通过率
  → 通过 → 选定，结束
  → 不通过 → 进入 Step 2

Step 2：往上加一档（Sonnet / GPT-5.1 / Gemini 3 Pro）
  → 跑同一组 eval
  → 通过 → 选定
  → 不通过 → 进入 Step 3

Step 3：用最强档（Opus 4.7 / GPT-5.5）
  → 跑同一组 eval
  → 通过 → 选定（但要算账，可能商业上不成立）
  → 不通过 → 这个需求当前模型做不到，砍掉或换路径
```

这个流程能让你**80% 的场景用 Haiku 级模型解决**——成本是 Opus 的 1/5。

## 「成本降 90% 效果只降 3%」真实案例

### 案例 1：Anthropic prompt caching 砍 90% 成本

DEV.to 一个作者用 prompt caching 把 RCA（根因分析）任务成本砍 90%——单次 RCA 从 $0.0065 降到千分之一美元级。

### 案例 2：月账单从 $720 → $72

Du'An Lightfoot 实测：仅靠加 cache_control，月账单从 $720 降到 $72，**效果无感知差异**。

### 案例 3：缓存命中率 7% → 74%

ProjectDiscovery 把动态内容挪出 cacheable prefix 后，命中率从 7% 提到 74%，账单降 59%。

**3 个案例的共同点**：模型没换、prompt 没改、效果不变——**只是 cache 用法变了，账单降 60%-90%**。

如果你还没系统用 prompt caching，**这是最低垂的果实**。

## 下半篇 · engineering reality

### 4 个维度的实战陷阱

#### 成本陷阱：算漏 retry 和上下文累积

PM 算成本最常漏 3 项：
1. **retry 成本**：失败 retry 1 次 = 成本翻倍
2. **上下文累积**：20 步 agent loop 单步 1K token 输出，累计 input 不是 20K 而是 **210K**（二次方膨胀）
3. **新 tokenizer 多吃 token**：Claude Opus 4.7 价格未变，但**新 tokenizer 同样文本要多吃 35% token**——账单不变 ≠ 账单不涨

#### 延迟陷阱：看 P50 没看 P99

PM 拍胸脯"这个模型延迟 1 秒"——指的是 P50。

实际生产环境用户感受是 P99——**最慢的 1% 请求**。

P99 通常是 P50 的 3-5 倍。如果你的业务场景有"对话式交互"——用户每 5-10 次对话就会遇到 1 次 P99 卡顿。**那 1% 决定用户口碑**。

选型时必须问 P99，不能只看 P50。

#### 效果陷阱：demo 效果 ≠ 生产效果

OpenAI / Anthropic / Google 的 demo 视频都是精挑细选的 case。

你的真实业务场景可能根本不在他们 demo 的能力范围内。

**唯一可信的效果数据是——你自己的 eval 集**。50-100 个真实任务，3 个独立标注员打分，**用置信区间报告结果**。Anthropic 的统计学方法论文专门讲这个：0.85 vs 0.87 在小样本下可能不显著。

#### 合规陷阱：数据出境 + 训练数据来源

B 端项目最容易翻车的是合规。

3 个必问：
- **数据是否出境**：DeepSeek V3.2 / Kimi K2.6 在中国境内推理，没有 BAA。Claude / GPT 数据可能跨境
- **训练数据来源**：客户合同里有没有"不能用客户数据训练"条款？需要 zero-data-retention（ZDR）模式。Anthropic 提供 ZDR，OpenAI 企业版默认 ZDR
- **企业级合规认证**：SOC 2 / HIPAA BAA / ISO 27001——B 端客户审计必查

如果你的项目涉及医疗 / 金融 / 政务——**合规维度可能直接淘汰一半模型**。这不是技术决策，是法务决策。

### Anthropic prompt caching 决策线

什么时候上 prompt caching？

> 当你的 system prompt **> 1024 token** 且复用率 **> 2 次** 时，cache 就回本。

Haiku 写 cache $3.75/M、读 $0.30/M——2 次即回本。Sonnet / Opus 类似。

**这条线意味着**：大部分有 system prompt 的产品都应该上 cache。

如果你的产品 system prompt 是 5000 token、日均调用 10 万次、复用率 100%——
- 不开 cache 月成本 ≈ ¥45 万
- 开 cache 月成本 ≈ ¥4.5 万

**差 10 倍**。

## ⚠️ Tokenizer 陷阱：Opus 4.7 价格未变，账单涨 35%

最容易被忽视的陷阱。

Claude Opus 4.7 标价没变（$5/$25 输入输出），但 Anthropic **换了新 tokenizer**——同样文本要多吃 35% token。

意思是：你的账单**自动涨 35%**，但定价页看不出来。

Finout 专门写过文章拆解：「The Real Cost Story Behind the Unchanged Price Tag」。

**给 PM 的建议**：模型升级时**实测 token 用量**，不要只看价格表。

## 质量工程视角看模型选型

**8 年做软件质量工程的视角**——

模型选型本质是「**多约束下的工程权衡**」——成本约束 + 延迟约束 + 效果约束 + 合规约束。

传统软件工程里有一个反复验证的原则：**「Premature optimization is the root of all evil」**——过早优化是万恶之源。

放到 AI 产品上——「**Premature 'use the strongest model' is the root of token bills out of control**」——过早选最强模型是 token 账单失控的根源。

**正确动作**：先用最便宜的模型跑通 PMF，确认能力上限够用，再按需升级。

这不是省钱小聪明，是质量工程的基本动作——**避免过早承担不必要的复杂度（成本是一种复杂度）**。
