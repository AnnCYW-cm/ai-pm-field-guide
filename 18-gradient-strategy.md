# 18 · AI 产品的灰度策略——传统 A/B 失效之后

> 第一季《AI 时代 PM 工作流重构》· 第 18 篇 · 第四部分 交付
> 预估字数：5000

---

第十七个"不以为然"——

**90% 的 PM 还在用「传统 A/B 测试」做 AI 灰度**。

不是 PM 不懂 A/B。是大多数 PM **以为"AI 产品的灰度和传统功能灰度一样"**——结果一个 prompt 版本灰度 50% 用户，3 天后发现 P99 延迟翻了 3 倍——**这是传统 A/B 不会 catch 的东西**。

读完这一篇你会知道——**AI 产品的灰度不是 A/B 分组，是 prompt 版本 / 模型版本 / 参数版本的多轴切换**。

---

## 上半篇 · workflow

### AI 灰度的 3 轴

传统 A/B 测试是单轴——A 组 vs B 组。

AI 产品灰度是**多轴**——至少 3 轴：

#### 轴 1：prompt 版本
- system prompt 改了
- few-shot examples 加了 / 删了
- 输出格式变了

#### 轴 2：模型版本
- 从 GPT-5 切到 GPT-5.1
- 从 Sonnet 切到 Opus
- 从 Claude 切到 Gemini

#### 轴 3：参数版本
- temperature 改了
- max_tokens 改了
- top_p 改了

**3 轴交叉**意味着你可能同时在跑：
- prompt v3 + 模型 GPT-5.1 + temperature 0.7
- prompt v3 + 模型 GPT-5.1 + temperature 0.5
- prompt v3 + 模型 Sonnet 4.5 + temperature 0.7
- prompt v4 + 模型 GPT-5.1 + temperature 0.7
- ...

如果不管理好——**组合爆炸**。

### 多轴灰度的设计原则

#### 原则 1：每次只动 1 轴
- 改 prompt 时不要同时换模型
- 换模型时不要同时调 temperature
- 否则出了问题不知道哪个变量导致

#### 原则 2：每个变化都跑 shadow eval
- 影子流量复制一份给候选版本
- 候选版本输出只进日志不返回用户
- 用 LLM judge 对比新旧版本

#### 原则 3：每次灰度都跑 regression eval
- 历史 case 集必须不退化
- 防止"修了 A 坏了 B"

#### 原则 4：灰度比例小步走
- 5% → 20% → 50% → 100%
- 每一步至少观察 24 小时

### 灰度评估指标

不只看转化，**还要看质量分布**：

| 指标维度 | 看什么 | 阈值 |
|---------|-------|------|
| 业务指标 | 转化率 / 留存 / NPS | 灰度组 ≥ 对照组 |
| 质量指标 | eval 通过率 / bad case 比例 | 灰度组 ≥ 对照组 |
| 延迟指标 | P50 / P99 | 灰度组 P99 ≤ 对照组 1.2x |
| 成本指标 | 单次调用成本 / 月度预算 | 灰度组 ≤ 对照组 1.1x |

**关键反共识**：质量分布要看分布，不能只看均值。

均值 90% 准确率掩盖了"5% 的 case 准确率 < 50%"。**那 5% 决定用户口碑**。

### 灰度失败的回滚

传统功能灰度失败 → 切回旧版本。

AI 产品灰度失败 → 不仅切回旧版本，**还要回滚 eval 基线**。

为什么？因为新版本期间收集的真实流量可能已经污染了 eval 集——你以为的"基线"已经不是基线。

**回滚 SOP**：
1. 切流量回旧版本（< 5 分钟）
2. 标记本次灰度期间的 eval 数据（隔离）
3. 重新跑历史 eval 集（验证基线没漂移）
4. postmortem（24 小时内）

## 业内多轴灰度具体实践

### 实践 1：Notion AI 双轨 eval ⭐

70 名工程师，**两套 eval suite**：
- **regression eval**：防止退步
- **frontier eval**：发现新模型在哪些子任务上比旧模型强

新前沿模型发布几小时内就能定位"这个模型在哪类 case 上值得切"。

issue 处理速度从每天 3 个提到 **30 个**——10 倍效率。

### 实践 2：DoorDash 客服 chatbot 模拟器

做法：
- 用历史 transcript 生成多轮合成对话
- 几分钟跑数百场
- 用 LLM judge 打分

线上之前**显著降低幻觉率**（来自 DoorDash 工程博客公开案例，具体降幅以官方为准）。

### 实践 3：Shadow traffic 业内通用实施

做法：
- 生产请求复制一份给候选模型
- 候选输出只进日志不返回用户
- 用 LLM judge 比对、统计 token / latency / cost

**代价**：API 成本翻倍（必须设预算告警）。

### 实践 4：Canary 5-10% + 自动卡 PR

做法：
1. 改 prompt 的 PR 自动跑 eval，分数低于基线则 block 合并
2. canary **5-10% 真实流量**
3. prompt 版本不可变，留 instant rollback
4. 生产抽样实时和 staging 比对

### 实践 5：Hamel Husain Eval-Driven Workflow

SOP：
1. 先看 100 条真实 trace 做 error analysis
2. 聚类失败模式
3. 写 eval（不是先写 eval 再发现问题）
4. 每次改 prompt 跑全量回归

**Hamel 原则**："**until you stop finding new failure modes**"（在没新失败模式之前不要停——Hamel 强调"看到没新类别才停"，不是固定看几条）。

## 下半篇 · engineering reality

### 影子流量（shadow traffic）的工程实现

为什么 AI 产品比传统产品更需要影子流量？

因为 AI 输出是**概率分布**——你光看新版本的 eval 通过率不够，必须看它在**真实流量分布上**的表现。

shadow traffic 实现要点：
- **异步执行**：不阻塞用户请求
- **采样而非全量**：通常 5-10% 流量做影子（成本可控）
- **结果只入日志**：不返回用户
- **对比维度**：质量 / 延迟 / 成本三维

### Shadow eval：不影响用户的前提下评估新版本

shadow eval 是 shadow traffic 的进阶——不仅记录新旧对比，**还跑 LLM judge 给两个版本打分**。

具体流程：
```
1. 用户请求进入系统
2. 主流量走旧版本（用户看到旧版本输出）
3. 复制一份请求异步发给新版本
4. 旧版本输出 + 新版本输出 → 一起进 LLM judge
5. LLM judge 打分：哪个版本更好
6. 累计 7 天数据 → 决定是否切流量
```

**关键**：LLM judge 必须先人工对齐（详见第 12 篇 Eval 章节）。

### Regression eval：每次灰度都要跑历史 case 集

为什么必须跑回归？

因为 AI 系统是**全局耦合的**——改了 A 部分的 prompt，可能让 B 部分跑得更差。

**真实案例 1：OpenAI GPT-4o sycophancy（2025-04）**

为了让对话"更友好"，结果训练成附和有害决策（甚至附和停药）。

**关键失败点**：sycophancy 没有作为 blocking 项进入发版评估。**几位测试员"感觉怪"但没机制让这条意见停下发版**。

修复后官方明确写入"behavior issues 必须能 block 发布"。

**真实案例 2：Gemini 图像生成 over-tune（2024-02）**

为了修"输出不够多元"，结果 over-tune 到画 1800 年代美国参议员、纳粹士兵都强行多元，且变得过度拒答。

CEO Pichai 公开发文称"unacceptable"。

### 一个灰度翻车案例：「单轴看效果好，多轴交叉后崩了」

业内案例：**ChatGPT「胡言乱语事件」（2024-02-20）**

一次"用户体验优化"上线引入 token 选择 bug，模型连续输出无意义重复（如 30 多遍 "a synonym for 'overgrown' is 'overgrown'"）、夹杂德语和"奇幻语言"。

**线上约 5 小时才回滚**（OpenAI status page 报告为 "several hours"）。

**教训**：UX 层的"小优化"也能炸掉模型层；任何改动都要走 shadow eval。

## 灰度策略的 4 阶段模型

```
阶段 1：内部测试（0% 真实流量）
  - 5-10 内测用户
  - 跑核心 eval 集
  - 通过 → 进阶段 2

阶段 2：Shadow eval（0% 用户可见）
  - 5% 真实流量复制到新版本
  - 异步执行 + LLM judge 对比
  - 累计 3-7 天数据
  - 新版本质量 ≥ 旧版本 → 进阶段 3

阶段 3：Canary 灰度（5% 用户可见）
  - 5% 真实流量看到新版本
  - 监控业务指标 + 质量指标 + 延迟指标 + 成本指标
  - 任一指标退化 → 立刻回滚
  - 通过 → 进阶段 4

阶段 4：分批放量（20% → 50% → 100%）
  - 每一步观察 24 小时
  - 期间持续跑 regression eval
  - 100% 后再观察 7 天再确认稳定
```

完整走完需要 **2-4 周**——这是 AI 产品负责任的灰度节奏。

## 质量工程视角看 AI 灰度 = 受控实验

**8 年做软件质量工程的视角**——

任何生产级系统的发版都遵循一个原则：「**受控实验，逐步放量**」。

传统软件遵循 canary deployment（金丝雀部署）—— 5% → 20% → 50% → 100%。

AI 产品在此之上还要加 2 个维度：
1. **shadow eval**——发版前 0% 用户可见的对比
2. **regression eval**——发版时回归测试老 case

为什么 AI 产品比传统软件多 2 个维度？因为：
- AI 输出是概率分布——单点对比不够
- AI 系统全局耦合——改 A 可能影响 B

「**4 阶段灰度**」本质是 AI 时代的 canary deployment——多出 shadow 和 regression 两层。

## 给你的可执行清单

#### 动作 1：你下一次发版必须按 4 阶段模型走
- 内部测试 → Shadow eval → Canary 5% → 分批放量

#### 动作 2：3 轴灰度严格分离
- 每次只动 1 轴
- prompt / 模型 / 参数 不要同时改

#### 动作 3：所有改动跑 regression eval
- 历史 case 集必须不退化
- PR 自动 block 退化的改动

#### 动作 4：建立 shadow traffic 基础设施
- 5-10% 流量异步复制
- LLM judge 对比新旧版本
- 至少累计 3-7 天数据再做决策
