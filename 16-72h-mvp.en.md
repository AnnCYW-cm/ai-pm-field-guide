The fifteenth "dismissive blind spot" —

**90% of PMs are still building "a smaller version of the full product" as the MVP**.

It's not that PMs don't know the MVP concept. It's that most PMs **assume "an MVP is the full product with 80% sliced off"** — they spend a month and still haven't shipped, every capability is a thin scratch on the surface, **and when they demo it the boss has no idea what they're looking at**.

By the end of this chapter you will know — **an AI product's MVP isn't "a smaller product"; it's "the smallest unit that demonstrates a capability loop". 72 hours is a ceiling, not a target.**

## Part 1 · Workflow

### The 72-Hour Breakdown

#### Day 1: PRD + prototype (8 hours)

```
2 hours: write the "Product Note" (Anthropic style, 3-4 paragraphs)
  - user intent (1 paragraph)
  - desired outcome (1 paragraph)
  - concrete metrics (1-2 paragraphs)

2 hours: write the Eval set (20-50 cases)
  - 3 evaluation dimensions
  - pass-line

2 hours: draw the capability flow
  - 3-5 core capabilities
  - dependencies + fallback paths

2 hours: pick the model + run capability probes
  - 4-dimension comparison (cost / latency / quality / compliance)
  - run 5-10 real cases to verify feasibility
```

**End-of-Day 1 deliverable**: a runnable PRD skeleton + Eval set + model selection decision.

#### Day 2: Model integration + core capability (16 hours)

```
4 hours: scaffolding (pick one of Lovable / Bolt / v0 / Claude Code)
  - use a vibe-coding tool (Chapter 17 deep-dive coming)
  - don't write from scratch

8 hours: integrate the core capability
  - 1-2 core capabilities running (not all)
  - bare-bones UI (don't beautify)
  - no production DB / no auth / no payments

4 hours: run the Eval set
  - check pass rate
  - tag bad cases
  - decide go / no-go
```

**End-of-Day 2 deliverable**: a prototype that runs the core capability loop end-to-end.

#### Day 3: Internal test + demo (8 hours)

```
2 hours: get 3-5 target users to try it
  - observe behavior, don't collect wishlists
  - note "where they got stuck"
  - find cases outside the happy path

3 hours: fix critical bugs
  - only fix Tier-4 catastrophic + Tier-3 blocking
  - don't fix Tier 1-2 (it's a demo, doesn't matter)

3 hours: record the demo
  - Loom, 3-5 minute video
  - 1-page Notion ROI walk-through
  - no formal deck needed
```

**End-of-Day 3 deliverable**: a demoable demo + screen recording + decision doc.

### The Audience Determines the MVP Form

Three different MVP forms for three different audiences:

#### Form 1: For the boss
- **Goal**: get project approval / continued investment decision
- **Form**: Loom recording + 1-page Notion ROI walk-through
- **Don't do**: fancy UI / a complete product
- **Focus**: help the boss see "why AI can do this"

#### Form 2: For customers
- **Goal**: collect LOIs / paid intent
- **Form**: Lovable / Bolt deployed demo link + waitlist
- **Don't do**: real data / user auth
- **Focus**: let the customer click and feel the core capability

#### Form 3: For team alignment
- **Goal**: build shared understanding / resolve review disagreements
- **Form**: v0-generated UI + longest happy path runnable
- **Don't do**: complete workflow / multi-role permissions
- **Focus**: let the team "see the same thing" before discussing

## Part 2 · Engineering Reality

### The 5 things an MVP shouldn't do

#### 1. A complete product
An MVP isn't a product — it's a **capability validation**.

A complete product needs: auth / payments / email / push notifications / customer support / compliance — none of which fit in 72 hours.

**Mitigation**: **MVPs don't touch real production.** Auth = hardcoded users, payments = fake data, email = console.log, support = skip.

#### 2. High-quality UI
A pretty UI makes both PM and engineer think "this is almost ready to ship" — **a dangerous illusion**.

The MVP exists to "see if the core capability runs" — the UI should be **deliberately ugly** so everyone focuses on the core capability.

**Mitigation**: default Tailwind styling / shadcn components / system fonts. Don't spend time on mockups.

#### 3. Full eval
At MVP stage, vibe-check is enough (look at 10-20 cases yourself).

Formal eval (50-200 cases + multi-rater labeling + pass-line) waits until pre-launch.

**Mitigation**: write a lightweight eval (10 cases), confirm it runs. Formal eval comes in week 3.

#### 4. Performance optimization
At MVP stage, don't optimize latency, don't optimize token cost, don't optimize concurrency.

All of that is pre-launch work. At MVP stage you're verifying "**the core capability works**", not "**the core capability is fast**".

**Mitigation**: use the most expensive model (e.g. Opus 4.7) to confirm quality. Downgrade to Haiku-class cheaper models before launch.

#### 5. Complete error handling
At MVP stage, let it crash — read the trace and fix it yourself.

Full error handling (try/catch / fallback / retry) waits until pre-launch.

**Mitigation**: write only the happy path. Don't handle the 99% of edge cases and errors.

### The Chasm Between "Demoable" and "Launchable"

An MVP cannot become a product directly. **There is at least 2x more work between them.**

| Dimension | Demoable MVP | Launchable product |
|-----------|--------------|--------------------|
| Users | 5-10 internal testers | 1000+ real users |
| Concurrency | 1 | 100-1000 |
| Error handling | none | complete fallback |
| Data | mock | production DB |
| Auth | fake login | real authentication |
| Compliance | not considered | GDPR / data isolation |
| Monitoring | none | dashboard + alerts |
| Rollout | none | multi-axis gradual + shadow eval |
| Eval | vibe check | full eval + CI |
| Cost control | not considered | hard cap + breaker |

**From MVP to launchable product: at least 4–8 weeks** — not "72 hours doubled".

The PM must understand this. Don't see the MVP and tell the boss "we can launch next week" — **that is bluffing**.

### Managing the 72-Hour Price Tag

**Any requirement that can't produce an MVP within 72 hours is fundamentally an un-converged requirement.**

Why? Because 72 hours is the critical point for vibe-coding tools:
- < 24 hours: single capability validation
- 24-72 hours: core capability loop
- > 72 hours: you're not building an MVP, you're building a product

If you exceed 72 hours — go back and re-examine the requirement:
- Is the requirement too big? Split into 2-3 MVPs
- Wrong path? Prompt → Agent → Fine-tune
- Wrong model? Re-run capability probes and re-select
- Wrong tool? Switch vibe-coding tools

**Don't grind through it.** A grinded-out MVP is 99% pointing the wrong direction.

## QE Lens: MVP = Minimum Falsifiable Unit

**8 years of software quality engineering perspective** —

In quality engineering, MVP has a more precise name — **Minimum Falsifiable Unit**.

The core of the scientific method is "**propose hypothesis → design experiment → falsify**". The MVP is the "design experiment" step.

**The key to experiment design is not "produce a complete product"; it's "rapidly falsify the key hypotheses"** —
- Hypothesis 1: users will use AI to solve this problem (falsifier: do users actually use the demo?)
- Hypothesis 2: the model can stably output acceptable results (falsifier: does eval clear the pass line?)
- Hypothesis 3: the business model closes the loop (falsifier: is the unit cost math clean?)

If any one of these three hypotheses is falsified — **the product shouldn't be greenlit**.

72 hours = the shortest cycle to falsify those 3 hypotheses. Beyond 72 hours = you're no longer running a falsification experiment, you're building a product.
